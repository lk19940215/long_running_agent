# 05 - 上下文压缩

> 一句话：5 层压缩管道——从最便宜的裁剪到最昂贵的全量摘要——逐层升级，让无限长的对话塞进有限的上下文窗口。

## 为什么上下文压缩是核心？

Claude 的上下文窗口有上限（200K token）。一次深度编码对话很容易产出数十万 token 的工具调用结果、代码片段、文件内容。**没有压缩，对话几轮就会爆掉**。

上下文压缩不是简单的"截断"——它要在**保留关键信息**（当前任务、已修改的文件、关键决策）和**控制大小**之间找平衡。Claude Code 为此设计了 5 层递进策略。

---

## 全景图

```
每轮 query 循环开始时
  ↓
  原始消息序列（可能数百条、几十万 token）
  ↓
  ① 工具结果预算    applyToolResultBudget()
  ↓  超大工具输出 → 持久化到磁盘 → 替换为摘要+路径
  ↓
  ② Snip 裁剪       snipCompactIfNeeded()            [HISTORY_SNIP]
  ↓  按标记裁剪旧消息 → 占位符替换
  ↓
  ③ 微压缩          microcompactMessages()
  ↓  移除过时工具结果的详细内容 → 占位文本
  ↓  或 cache_edits → API 层直接删除（不改本地消息）
  ↓
  ④ 上下文折叠      applyCollapsesIfNeeded()          [CONTEXT_COLLAPSE]
  ↓  已完成的段落 → 折叠为摘要（保留细粒度上下文）
  ↓
  ⑤ 自动压缩        autoCompactIfNeeded()
  ↓  token > 阈值 → 全量摘要（调用模型生成摘要）
  ↓
  压缩后的消息 → 送给 Claude API
  
  ↓ （如果 API 返回 413 / 媒体错误）
  ↓
  ⑥ 响应式压缩      tryReactiveCompact()             [REACTIVE_COMPACT]
     紧急压缩 → 剥离超大内容 → 重试
```

---

## 为什么是这个顺序？

```
便宜 ←────────────────────────→ 昂贵
快速 ←────────────────────────→ 慢速
保留细节 ←────────────────────→ 丢失细节

① 工具预算   ② Snip   ③ 微压缩   ④ 折叠   ⑤ 自动压缩   ⑥ 响应式
  ↑ 本地替换    ↑ 标记裁   ↑ 清工具    ↑ 段落摘要  ↑ 全量摘要    ↑ 紧急
  ↑ 零 API      ↑ 零 API   ↑ 零/低     ↑ 低 API    ↑ 高 API     ↑ 最后手段
```

**核心原则**：先用便宜手段（纯本地操作），不够再升级到调 API 的手段。如果前面的层够用了，后面就跳过。

---

## ① 工具结果预算 — applyToolResultBudget()

**问题**：单个工具输出可能几十万字符（比如 `grep` 搜全仓库）。

**解法**：每条 `tool_result` 有大小预算（`maxResultSizeChars`），超出的：
1. 持久化到磁盘文件
2. 消息中替换为「结果已保存到 /path/to/file，前 N 行预览如下...」

**触发**：每轮循环开头，在所有压缩之前。

**特殊工具**：`Read` 工具的 `maxResultSizeChars` 设为 `Infinity`——持久化会导致"Read → 文件 → Read"的无限循环。

---

## ② Snip 裁剪 — snipCompactIfNeeded()

**Feature Gate**：`HISTORY_SNIP`

**问题**：对话历史中很多早期消息已经完全过时（比如最初的探索、已解决的问题），但 autocompact 要等到阈值才触发。

**解法**：在消息中插入 snip 标记，标记之前的消息被替换为占位符。

```
消息 1: 用户问了 A（已解决）      → [snipped]
消息 2: 模型回答 A               → [snipped]
消息 3: 用户问了 B（已解决）      → [snipped]
消息 4: 模型回答 B               → [snipped]
── snip 标记 ──
消息 5: 用户问了 C（当前话题）    → 保留
消息 6: 模型回答 C               → 保留
```

**优点**：零 API 调用，纯本地操作。
**缺点**：丢失早期上下文的细节（但对当前任务通常无影响）。

**与 query.ts 的交互**：`snipTokensFreed` 参数传给后续的 autocompact，修正 token 估算。

> ⚠️ **注意**：本还原版中 `snipCompact.ts` 是占位实现（直接返回原消息）。完整逻辑在 Anthropic 内部构建中。

---

## ③ 微压缩 — microcompactMessages()

**文件**：`services/compact/microCompact.ts`

**问题**：工具调用的详细输入/输出在后续对话中通常不再需要（模型已经"看过"了），但它们占大量 token。

**两种模式**：

### 模式 A：时间触发微压缩

当检测到**长时间间隔**（用户离开后回来），把旧的可压缩工具结果替换为占位文本：

```
原始：Read { file: "src/main.tsx" } → 4700 行完整代码
替换：[Tool result cleared - conversation resumed after 15 min gap]
```

**可压缩工具列表**（`COMPACTABLE_TOOLS`）：
`Read`, `Bash`, `Grep`, `Glob`, `WebSearch`, `WebFetch`, `Edit`, `Write`

### 模式 B：缓存编辑微压缩（Cached Microcompact）

**Feature Gate**：`CACHED_MICROCOMPACT`

利用 API 的 `cache_edits` 功能，在**服务端**直接删除工具结果的缓存条目，本地消息不变：

```
本地消息不变（仍有完整工具结果）
API 请求携带 cache_edits: [{ type: 'delete', tool_use_id: 'xxx' }]
→ 服务端忽略这些 tool_result 的 token 计数
→ 实际上下文缩小，但本地恢复（resume）时数据完整
```

**优点**：
- 零信息丢失（本地完整保留）
- 利用 prompt cache（同一轮的后续请求仍能命中缓存）

**关键常量**：`IMAGE_MAX_TOKEN_SIZE = 2000`

---

## ④ 上下文折叠 — applyCollapsesIfNeeded()

**Feature Gate**：`CONTEXT_COLLAPSE`

**问题**：Autocompact 把整段对话压成一段摘要，丢失了中间的结构。如果只是某些"段落"过时了，全量压缩太浪费。

**解法**：把已完成的"段落"（一组相关的对话轮次）折叠为摘要，保留其他段落的原始细节。

```
段落 1（探索阶段）→ 摘要："分析了 3 个方案，选定方案 B"
段落 2（实现阶段）→ 保留完整对话
段落 3（测试阶段）→ 保留完整对话
```

**与 Autocompact 的关系**：
- 折叠在 autocompact **之前**执行
- 如果折叠够了（token 降到阈值以下），autocompact 就不触发
- 这保留了更多细粒度上下文

**在 query.ts 中的位置**：

```
④ applyCollapsesIfNeeded()    ← 先折叠
⑤ autoCompactIfNeeded()       ← 折叠不够才全量压缩
```

**413 恢复**：当 API 返回 prompt-too-long 时，`recoverFromOverflow()` 排空所有已暂存的折叠。

> ⚠️ **注意**：本还原版中 `contextCollapse/index.ts` 是占位实现（`isContextCollapseEnabled()` 恒 `false`）。

## ⑤ 自动压缩 — autoCompactIfNeeded()

**文件**：`services/compact/autoCompact.ts`

**这是最重要的压缩策略**——当 token 超过阈值时，调用模型生成整段对话的摘要。

### 阈值计算

```
模型上下文窗口（如 200K）
  - 预留输出 token（min(模型 max_output, 20000)）
  = 有效上下文窗口
  - 自动压缩缓冲（13000）
  = 自动压缩阈值            ← token 超过这个值就触发
  
  - 手动压缩缓冲（3000）
  = 阻塞限制                ← 超过这个值连手动 /compact 的空间都没了
```

### 执行流程

```
autoCompactIfNeeded()
  ↓ 检查 token 是否超过阈值
  ↓ 检查是否被禁用（DISABLE_COMPACT / DISABLE_AUTO_COMPACT）
  ↓ 检查熔断器（连续失败 >= 3 次则停止）
  ↓
  ↓ 优先尝试 Session Memory Compact
  ↓   ↓ 成功？→ 使用磁盘上的 session memory 替代模型摘要（零 API）
  ↓   ↓ 失败？→ 继续
  ↓
  ↓ compactConversation()
  ↓   ↓ 跑 pre-compact hooks
  ↓   ↓ 构造压缩提示（getCompactPrompt）
  ↓   ↓ 调用模型生成 <analysis> + <summary>
  ↓   ↓ 清 read cache
  ↓   ↓ 生成 post-compact 附件（最近读文件、计划、技能）
  ↓   ↓ 跑 post-compact hooks
  ↓   ↓ 返回 CompactionResult
  ↓
  ↓ buildPostCompactMessages()
  ↓   ↓ compact_boundary 系统消息
  ↓   ↓ 用户摘要消息（"会话从先前对话延续"）
  ↓   ↓ 附件消息（文件恢复、计划、技能）
  ↓
  → 压缩后的消息序列（通常从几百条 → 几条）
```

### compactConversation() 内部

```
输入：完整消息序列 + 系统提示 + 配置
  ↓
  构造压缩请求：
    系统提示 = "你是一个对话摘要助手"
    用户消息 = getCompactPrompt() + 对话内容
  ↓
  优先 fork 复用 prompt cache（避免重新计算系统提示 token）
  失败则流式调用
  ↓
  模型输出 <analysis>分析</analysis><summary>摘要</summary>
  ↓
  formatCompactSummary() → 提取 summary，去掉 analysis
  ↓
  生成 post-compact 附件：
    - 最近 5 个已读文件的内容快照（上限 5000 token/文件，总计 50000 token）
    - 当前计划（如果有）
    - 相关技能（上限 25000 token）
    - Delta 记忆（如有变化的 CLAUDE.md）
  ↓
  CompactionResult {
    summaryMessages, attachments, hookResults,
    preCompactTokenCount, postCompactTokenCount,
    compactionUsage
  }
```

### 关键常量

| 常量 | 值 | 说明 |
|------|-----|------|
| `AUTOCOMPACT_BUFFER_TOKENS` | 13,000 | 触发阈值缓冲 |
| `MAX_CONSECUTIVE_AUTOCOMPACT_FAILURES` | 3 | 熔断：连续失败 3 次停止 |
| `POST_COMPACT_MAX_FILES_TO_RESTORE` | 5 | 压缩后恢复的文件数 |
| `POST_COMPACT_TOKEN_BUDGET` | 50,000 | 压缩后附件总 token 预算 |
| `POST_COMPACT_MAX_TOKENS_PER_FILE` | 5,000 | 单文件 token 上限 |
| `MAX_PTL_RETRIES` | 3 | 压缩请求自身 PTL 时的裁头重试 |
| `MAX_COMPACT_STREAMING_RETRIES` | 2 | 流式回退路径重试 |

### Session Memory Compact

**优先于** `compactConversation()`，利用磁盘上已有的 session memory：

```
条件：session_memory 功能启用 + sm_compact 功能启用
  ↓ 从磁盘加载 session memory（之前的对话摘要）
  ↓ 构造 CompactionResult（不调模型）
  → 零 API 调用的压缩
```

**配置**：
- `minTokens: 10_000`（至少要压缩这么多才值得）
- `minTextBlockMessages: 5`（至少有 5 条含文本的消息）
- `maxTokens: 40_000`（session memory 摘要的 token 上限）

## ⑥ 响应式压缩 — tryReactiveCompact()

**Feature Gate**：`REACTIVE_COMPACT`

**问题**：有时 token 估算不准确，autocompact 判断"没超阈值"但 API 返回 413（prompt-too-long）。

**解法**：在 API 返回 413 后**紧急压缩**，然后重试。

### 在 query.ts 中的位置

```
API 流式响应...
  ↓ 收到 413 错误
  ↓ 扣留（不 yield 给外层）
  ↓
  先尝试 Context Collapse 排空
  ↓ 没用？
  ↓
  tryReactiveCompact()
  ↓ 压缩成功 → state = next → continue（重试）
  ↓ 压缩失败 → yield 错误 → 退出
```

**触发条件**：
1. **prompt-too-long (413)**：上下文超过模型窗口
2. **media-size-error**：图片/PDF 太大
3. 尚未尝试过响应式压缩（`hasAttemptedReactiveCompact` 守卫，防止死循环）

> ⚠️ **注意**：本还原版中 `reactiveCompact.ts` 是占位实现。

## 压缩管道在 query.ts 中的位置

```typescript
// queryLoop 内 while(true) 循环
while (true) {
  // ① 工具结果预算
  messagesForQuery = await applyToolResultBudget(messagesForQuery, ...)

  // ② Snip
  if (feature('HISTORY_SNIP')) {
    const snipResult = snipModule.snipCompactIfNeeded(messagesForQuery)
    messagesForQuery = snipResult.messages
    snipTokensFreed = snipResult.tokensFreed
  }

  // ③ 微压缩
  const mcResult = await deps.microcompact(messagesForQuery, ...)
  messagesForQuery = mcResult.messages

  // ④ 上下文折叠
  if (feature('CONTEXT_COLLAPSE') && contextCollapse) {
    const collapseResult = await contextCollapse.applyCollapsesIfNeeded(messagesForQuery, ...)
    messagesForQuery = collapseResult.messages
  }

  // ⑤ 自动压缩
  const { compactionResult } = await deps.autocompact(messagesForQuery, ...)
  if (compactionResult) {
    messagesForQuery = buildPostCompactMessages(compactionResult)
  }

  // ... 调 API → 流式接收 → 工具执行 ...

  // ⑥ 响应式压缩（仅在 413/媒体错误后触发）
  if (isWithheld413 && reactiveCompact) {
    const compacted = await reactiveCompact.tryReactiveCompact(...)
    if (compacted) { state = next; continue }
  }
}
```

## 压缩策略对比

| 策略 | API 调用 | 信息丢失 | 触发条件 | Feature Gate |
|------|---------|---------|---------|-------------|
| **① 工具预算** | 零 | 低（保留预览） | 每轮自动 | 无 |
| **② Snip** | 零 | 中（旧消息丢失） | 有标记时 | `HISTORY_SNIP` |
| **③ 微压缩** | 零/低 | 低（只清工具结果） | 每轮自动 | 无 / `CACHED_MC` |
| **④ 折叠** | 低 | 中（段落摘要） | 有可折叠段落时 | `CONTEXT_COLLAPSE` |
| **⑤ 自动压缩** | **高** | **高**（全量摘要） | token > 阈值 | 无 |
| **⑥ 响应式** | **高** | **高** | API 413 错误 | `REACTIVE_COMPACT` |

## 扣留-恢复模式（Withhold）

这是 query.ts 中最精巧的设计之一，值得单独说明：

```
API 返回可恢复的错误（413 / max_output_tokens / media_size）
  ↓
  不立即 yield 给外层（"扣留"）
  ↓ 放入 assistantMessages[] 但不 yield
  ↓
  尝试恢复：
  ├─ 413 → Context Collapse 排空 → Reactive Compact
  ├─ max_output → 提升限制（8K→64K）→ 注入续写提示
  └─ media → Reactive Compact 剥离
  ↓
  恢复成功？→ 丢弃扣留的错误消息 → state = next → continue
  恢复失败？→ yield 扣留的错误消息 → 退出
```

**为什么不直接 yield？** 因为 SDK 调用方（如 desktop app）收到 error 消息后可能立即终止会话。如果直接 yield，恢复循环还在跑但没人听了。

## 设计亮点

| 亮点 | 说明 | 可借鉴场景 |
|------|------|----------|
| **5 层递进** | 从便宜到昂贵逐层升级，每层独立 | 任何需要多级缓存/压缩的系统 |
| **Session Memory 替代** | 用已有摘要替代再次调模型 | 重复性计算的缓存复用 |
| **Cache Edits** | API 层删除而非本地修改，保留恢复能力 | 需要软删除/懒删除的场景 |
| **熔断器** | 连续失败 3 次停止重试 | 任何自动重试系统 |
| **扣留-恢复** | 可恢复错误不暴露给调用方 | 流式系统的优雅降级 |
| **Feature Gate** | 每层独立开关，可逐步灰度 | 大型系统的渐进式发布 |
