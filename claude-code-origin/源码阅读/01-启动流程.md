# 01 - 启动流程

> 一句话：从 `bun run dev` 到用户看到交互界面，代码穿过 4 个文件，本质是"逐层加载、最后装配"。

## 为什么要先看启动流程？

启动流程是源码阅读的"地图入口"。搞清楚代码从哪里开始执行、经过哪些关键节点、最终在哪里"稳定下来"，后续才能有方向地深入各个模块。

## 全景图

```
bun run dev
  ↓
bootstrap-entry.ts          ← 5 行代码，最小入口
  ↓ ensureBootstrapMacro()  ← 挂载全局 MACRO（还原版兼容代码）
  ↓ await import('cli.tsx') ← 动态导入，确保 MACRO 先就位
  ↓
entrypoints/cli.tsx         ← 命令行参数路由分发器
  ↓ 环境准备（3 个顶层副作用）
  ↓ void main()             ← 模块 import 时自动执行
  ↓ 11 个快速路径 if 判断    ← 特殊命令提前 return
  ↓ (默认路径)
  ↓ await import('main.tsx')
  ↓ cliMain()
  ↓
main.tsx                    ← 4700 行编排中心
  ↓ main()
  ↓   安全防护 + 信号处理
  ↓   协议/深度链接处理
  ↓   客户端类型判断
  ↓   eagerLoadSettings()
  ↓   run()
  ↓
  ↓ run()
  ↓   Commander 创建 + 注册选项
  ↓   preAction: init() + 迁移 + 远端配置
  ↓   action: 2800 行编排 → setup() → 加载工具/Agent → launchRepl()
  ↓
REPL.tsx                    ← 交互式界面启动
```

## 逐站解析

## 第一站：bootstrap-entry.ts

**文件**：`src/bootstrap-entry.ts`（5 行）

```typescript
import { ensureBootstrapMacro } from './bootstrapMacro'
ensureBootstrapMacro()           // 挂载全局 MACRO 常量
await import('./entrypoints/cli.tsx')  // 动态导入 CLI
```

**要点**：
- 使用 `await import()` 而非静态 `import` → 确保 MACRO 先挂载完毕
- `bootstrapMacro.ts` 是还原版兼容代码（原版通过 Bun 编译时注入 MACRO）

### 站 2：cli.tsx — 命令行路由分发器（300 行）

## 第二站：bootstrapMacro.ts

**文件**：`src/bootstrapMacro.ts`（30 行）

**做了什么**：在 `globalThis` 上挂载 `MACRO` 对象

```typescript
(globalThis as ...).MACRO = {
  VERSION: pkg.version,      // 版本号
  BUILD_TIME: '',             // 构建时间
  PACKAGE_URL: pkg.name,      // 包 URL
  ...
}
```

**为什么用 globalThis？**

| 方式 | 原版 Anthropic | 还原版 |
|------|--------------|--------|
| 注入时机 | 编译时（Bun bun:bundle 宏） | 运行时 |
| 类比 | webpack DefinePlugin / Vite define | globalThis 手动挂载 |
| 效果 | 代码中 `MACRO.VERSION` 被替换为字面量 | 运行时从全局对象读取 |

`globalThis` 是 ES2020 标准，在任何 JS 环境下都指向全局对象：
- 浏览器 = `window`
- Node.js / Bun = `global`

## 第三站：entrypoints/cli.tsx

**文件**：`src/entrypoints/cli.tsx`（300 行）

**定位**：命令行参数路由分发器——整个应用的「大门」

### 结构
**做了什么**：根据命令行参数决定走哪条路径。
```
环境准备（顶层副作用）
  ↓
async function main() {
  const args = process.argv.slice(2)

  // 快速路径 1~11：特殊命令提前 return
  if (--version) { console.log(VERSION); return }
  if (--dump-system-prompt) { ... return }
  if (--chrome-mcp) { ... return }
  if (--daemon-worker) { ... return }
  if (remote-control) { ... return }
  if (daemon) { ... return }
  if (ps/logs/attach/kill) { ... return }
  if (new/list/reply) { ... return }
  if (environment-runner) { ... return }
  if (self-hosted-runner) { ... return }
  if (--worktree --tmux) { ... return }

  // ⭐ 默认路径：加载完整 CLI
  const { main: cliMain } = await import('../main.js')
  await cliMain()
}

void main()  // 模块 import 时自动执行
```

**快速路径分类**：

| 类别 | 命令 | 场景 |
|------|------|------|
| **用户日常** | `--version` | 查版本号（零模块加载） |
| **用户日常** | `--worktree --tmux` | git worktree 隔离分支 |
| **后台运行** | `ps/logs/attach/kill` | 后台会话管理 |
| **后台运行** | `daemon` | 长驻守护进程 |
| **后台运行** | `--daemon-worker` | 守护进程派生的 worker |
| **后台运行** | `new/list/reply` | 模板任务 |
| **后台运行** | `environment-runner` | BYOC 无头运行器 |
| **后台运行** | `self-hosted-runner` | 自托管运行器 |
| **浏览器集成** | `--claude-in-chrome-mcp` | Chrome 扩展 MCP 服务 |
| **浏览器集成** | `--chrome-native-host` | Chrome 原生消息主机 |
| **浏览器集成** | `--computer-use-mcp` | 计算机操作（屏幕/鼠标） |
| **远程/桥接** | `remote-control` | 本地暴露为远程环境 |
| **内部评测** | `--dump-system-prompt` | 导出系统提示词 |

### 设计理念
**所有 import 都是动态的 `await import()`**，这样：
- 每条路径只加载自己需要的模块
- `--version` 实现了**零额外模块加载**的极致性能
- 避免了 main.tsx 的 140 个 import 带来的启动延迟

## 第四站：main.tsx — 配置装配车间（4700 行）

**文件**：`src/main.tsx`（4700 行）

**做了什么**：把命令行参数、用户配置、工具列表、Agent 定义、MCP 服务器等所有东西组装好，构建出 `initialState`（AppState）和 `sessionConfig`（REPLProps），交给 `launchRepl()` 启动交互界面。

### 文件内部结构

```
1~200 行     导入层      140 个 import + 3 个顶层预加载副作用
200~580 行   辅助函数    遥测/迁移/预取等
585~862 行   main()      ← cli.tsx 入口，安全防护 + 协议处理 + 调 run()
890~3850 行  ⭐ run()    ← 真正的编排核心（下面展开）
3850~4500 行 子命令      mcp / auth / plugin / doctor / update 等
4500~4740 行 辅助函数    resetCursor / maybeActivateProactive 等
```

### main() → run() 调用关系

```
main()
  ↓ 信号处理（SIGINT / SIGTERM / SIGHUP）
  ↓ 协议/深度链接处理（claude:// URL scheme）
  ↓ 客户端类型判断（terminal / IDE / remote）
  ↓ eagerLoadSettings() — 提前加载用户设置
  ↓ run()
      ↓ Commander 程序创建 + 注册 40+ 命令行选项
      ↓ preAction 钩子（通用初始化）
      ↓ ⭐ action 处理器（2800 行）
      ↓ 子命令注册（mcp / auth / plugin / doctor 等）
```

### preAction 钩子 — 每个命令执行前的通用初始化

```
preAction(thisCommand)
  ↓ ensureMdmSettingsLoaded()       企业 MDM 设置加载
  ↓ ensureKeychainPrefetchCompleted() 钥匙串预读
  ↓ init()                          ⭐ 运行时初始化（配置/遥测/策略/MCP）
  ↓ initSinks()                     日志 sink 挂载
  ↓ runMigrations()                 一次性配置迁移
  ↓ loadRemoteManagedSettings()     企业远端托管配置（非阻塞）
  ↓ loadPolicyLimits()              策略限制加载（非阻塞）
```

### ⭐ action() — 七阶段编排

action() 是整个文件的灵魂，约 2800 行，按顺序分 7 个阶段：

#### A. 解析选项（~200 行）

从 Commander 解构出所有命令行参数：

```
debug, permissionMode, model, resume, mcpConfig,
systemPrompt, appendSystemPrompt, worktree, agent,
allowedTools, disallowedTools, printMode, maxTurns, ...
```

#### B. 配置初始化（~400 行）

| 配置项 | 做了什么 |
|--------|---------|
| 权限模式 | 根据 `--permission-mode` / `--dangerously-skip-permissions` 设置 |
| MCP 配置 | 加载 `--mcp-config` 文件，合并多个 MCP 服务器配置 |
| 工具过滤 | 根据 `--allowed-tools` / `--disallowed-tools` 过滤工具 |
| 模型选择 | 解析 `--model` 参数，处理别名（sonnet → 完整模型 ID） |
| 系统提示 | 加载 `--system-prompt` / `--system-prompt-file` |

#### C. setup() — 设置工作环境（~100 行）

```
setup()
  ↓ 确定工作目录 cwd（可能创建 git worktree）
  ↓ 分配会话 ID
  ↓ 启动 UDS 消息服务
  ↓ 注册内置插件和技能
```

**亮点**：`setup()` 与 `getCommands()` / `getAgentDefinitions()` **并行执行**（利用 IO 类型差异提速）。

#### D. 加载命令和 Agent（~100 行）

- 加载斜杠命令（/help、/model、/compact 等）
- 加载 Agent 定义（自定义 Agent + 内置 Agent）
- 合并工具列表（内置工具 + MCP 工具 + 插件工具）

#### E. 非交互模式（~500 行）— 仅 `-p` / `--print` 模式

不启动 REPL，直接：发请求 → 打印结果 → 退出。支持 text / json / stream-json 三种输出格式。

#### F. showSetupScreens()（~100 行）— 仅交互模式

首次使用时显示：信任对话框 → 权限确认 → 登录引导 → 版本检查。

#### G. ⭐ 场景分发 → launchRepl()（~1500 行）

这是 action() 的最终目标。根据不同场景构建 `initialState`（AppState）和 `sessionConfig`（REPLProps），然后调用 `launchRepl()`。

**核心数据流**：

```
命令行参数 ─┐
用户配置   ─┤
权限规则   ─┤
MCP 服务器 ─┼──→ initialState (AppState)  ──┐
模型设置   ─┤                               ├──→ launchRepl() → REPL.tsx
Agent 定义 ─┤                               │
工具列表   ─┘                               │
                                            │
commands ──────┐                            │
initialTools ──┤                            │
mcpClients ────┼──→ sessionConfig (REPLProps) ──┘
thinkingConfig ┘
```

**7 种场景**（都到达 `launchRepl()`，但 `initialState` 不同）：

| 场景 | 触发条件 | 区别 |
|------|---------|------|
| **普通启动** | 无特殊参数 | 默认 initialState，**99% 用户** |
| 恢复会话 | `--resume` / `--continue` | 从转录文件恢复消息和状态 |
| Direct Connect | `cc://` URL | 添加连接信息消息 |
| SSH 远程 | `ssh <host>` | 添加 SSH 会话信息 |
| 助手模式 | KAIROS | 设置 `kairosEnabled = true` |
| CCR 远程 | 容器环境 | 设置 `remoteSessionUrl`，过滤远程安全命令 |
| 带历史恢复 | `--resume` + 数据 | 恢复 initialState + 历史消息 |

### initialState（AppState）— 应用全局状态

`initialState` 是 `AppState` 类型，约 60+ 个字段，是 REPL 运行时的**全局状态容器**。在 main.tsx 中被构建，传入 `launchRepl()`，经由 `AppStateProvider`（React Context）注入到整个组件树。

| 类别 | 关键字段 | 说明 |
|------|---------|------|
| 模型 | `mainLoopModel`, `thinkingEnabled`, `effortValue`, `fastMode` | 模型选择和推理配置 |
| 权限 | `toolPermissionContext` | 工具权限规则（allow/deny/ask） |
| MCP | `mcp.clients`, `mcp.tools`, `mcp.commands` | MCP 服务器和工具 |
| 插件 | `plugins.enabled`, `plugins.errors` | 已加载插件 |
| Agent | `agentDefinitions`, `agentNameRegistry`, `tasks` | Agent 定义和任务 |
| 远程 | `replBridge*`, `remoteSessionUrl` | Bridge 和远程会话 |
| 文件 | `fileHistory`, `attribution` | 文件变更追踪和归因 |
| UI | `expandedView`, `footerSelection`, `spinnerTip` | 界面状态 |
| 推测 | `speculation`, `promptSuggestion` | 投机执行和提示建议 |

> **深入了解**：AppState 的完整结构、Store 实现、状态流转机制，详见 [03-状态管理.md](./03-状态管理.md)

### sessionConfig（REPLProps）— REPL 会话配置

`sessionConfig` 对应 REPL 组件的 Props，描述本次会话的**运行时配置**：

| 字段 | 说明 |
|------|------|
| `commands` | 可用的斜杠命令列表（内置 + MCP） |
| `initialTools` | 初始工具列表 |
| `mcpClients` | MCP 服务器连接 |
| `thinkingConfig` | 思考模式配置（enabled / adaptive / disabled） |
| `initialMessages` | 初始消息（恢复会话时有值） |
| `systemPrompt` / `appendSystemPrompt` | 自定义系统提示 |
| `mainThreadAgentDefinition` | 主线程 Agent 定义 |
| `remoteSessionConfig` / `sshSession` | 远程会话配置 |
| `onBeforeQuery` / `onTurnComplete` | 查询前后回调 |

### 第五站：launchRepl() → REPL.tsx

`launchRepl()` 本身只有 10 行（`replLauncher.tsx`）：

```typescript
export async function launchRepl(root, appProps, replProps, renderAndRun) {
  const { App } = await import('./components/App.js')
  const { REPL } = await import('./screens/REPL.js')
  await renderAndRun(root, <App {...appProps}><REPL {...replProps} /></App>)
}
```

它做的事：
1. 动态加载 `App`（AppStateProvider 包装器）和 `REPL` 组件
2. `App` 用 `initialState` 创建 `Store`，通过 React Context 注入全局状态
3. `REPL` 接收 `replProps`，启动交互界面
4. 从此进入 React 渲染循环 — **main.tsx 的终点就是 REPL 的起点**

后续的核心逻辑在 QueryEngine + query.ts 中。

## main.tsx 值得逐行深读吗？

**结论：不值得。** 理解结构即可，不需要读每一行。

### 3 个设计亮点（值得学习）

| 亮点 | 说明 | 可借鉴场景 |
|------|------|----------|
| 并行初始化 | `setup()` 与 `getCommands()` 并行（利用 IO 类型差异） | 任何有多个异步初始化的应用 |
| `feature()` 条件编译 | 构建时替换为 `true`/`false`，一套代码多版本 | 需要区分内部版/公开版的产品 |
| 分层加载 | 从 0 import → 按需 import → 完整 import 渐进式 | CLI 工具的启动性能优化 |

### 各模块深读价值排序

| 模块 | 价值 | 原因 |
|------|------|------|
| main.tsx | ⭐ | 配置装配，理解结构即可 |
| **QueryEngine.ts** | ⭐⭐⭐ | 会话级编排：多轮对话、上下文管理 |
| **query.ts** | ⭐⭐⭐⭐ | **核心循环**：消息组装 → API → 工具执行 → 循环 |
| **Tool.ts** | ⭐⭐⭐ | 工具定义/执行的核心抽象 |
| **context.ts** | ⭐⭐⭐ | 系统提示词组装 |
| **compact/** | ⭐⭐⭐ | 上下文压缩策略 |
