# 04 - 工具系统

> 一句话：`buildTool()` 定义工具 → `tools.ts` 注册到数组 → `query.ts` 调 API 获得 `tool_use` → `toolExecution.ts` 校验/权限/执行/映射结果 → 返回 `tool_result` → API 继续。

## 为什么要理解工具系统？

Claude Code 的核心能力——读文件、改代码、跑命令、搜索、网页浏览——全部由工具实现。模型（Claude API）本身只输出文本和 `tool_use` 请求，**工具系统把 AI 的意图变成真实的操作**。

---

## 全景图

```
Claude API 返回 tool_use block
  ↓ { name: "Edit", id: "xxx", input: { file_path, old_string, new_string } }
  ↓
query.ts
  ↓ toolUseBlocks[] 收集所有 tool_use
  ↓ 模型输出完毕
  ↓
toolOrchestration.ts — 编排执行
  ↓ partitionToolCalls() — 分区：只读并行 / 写操作串行
  ↓ runToolsConcurrently() 或 runToolsSerially()
  ↓
toolExecution.ts — 单工具执行管线
  ↓ findToolByName()         → 查找工具定义
  ↓ inputSchema.safeParse()  → Zod 校验输入
  ↓ validateInput()          → 额外校验（可选）
  ↓ runPreToolUseHooks()     → 预钩子（可修改/拦截）
  ↓ canUseTool()             → 权限检查（可能弹窗）
  ↓ tool.call()              → ⭐ 执行工具逻辑
  ↓ mapToolResultToToolResultBlockParam() → 格式化结果
  ↓ runPostToolUseHooks()    → 后钩子
  ↓
yield UserMessage { tool_result: { tool_use_id, content } }
  ↓
query.ts 把 tool_result 拼入消息 → 下一轮循环
```

## Tool 类型定义（Tool.ts）

### buildTool() — 工具工厂

所有工具都通过 `buildTool()` 创建，它用安全默认值打底，再用具体定义覆盖：

```typescript
const TOOL_DEFAULTS = {
  isEnabled: () => true,
  isConcurrencySafe: () => false,    // 默认不可并发（保守）
  isReadOnly: () => false,           // 默认非只读（保守）
  isDestructive: () => false,
  checkPermissions: () => ({ behavior: 'allow' }),
  toAutoClassifierInput: () => '',   // 默认不参与安全分类
  maxResultSizeChars: 30_000,
  // ...
}

function buildTool(def) {
  return { ...TOOL_DEFAULTS, userFacingName: () => def.name, ...def }
}
```

**设计原则**：**fail-closed**（默认保守），需要并发/只读/跳过权限的工具必须显式声明。

### Tool 接口的核心方法

| 方法 | 何时调用 | 做什么 |
|------|---------|--------|
| `call(args, context, canUseTool, msg, onProgress)` | 执行时 | ⭐ 工具的核心逻辑 |
| `prompt(options)` | 构建系统提示时 | 生成给模型的工具使用说明 |
| `description(input, options)` | 展示工具信息时 | 工具描述文本 |
| `checkPermissions(input, context)` | 执行前 | 权限检查 |
| `validateInput(input, context)` | 执行前 | 输入校验 |
| `inputSchema` | 执行前 | Zod v4 schema，校验参数 |
| `isReadOnly(input)` | 编排时 | 决定并行/串行 |
| `isConcurrencySafe(input)` | 编排时 | 决定是否可并发 |
| `isEnabled()` | 注册时 | 决定是否启用 |
| `mapToolResultToToolResultBlockParam(content, id)` | 执行后 | 把内部结果映射为 API 格式 |

### 工具结果类型

```typescript
type ToolResult<T> = {
  data: T                        // 工具的实际输出
  newMessages?: Message[]        // 可选：附加的消息
  contextModifier?: (ctx) => ctx // 可选：修改后续上下文
  mcpMeta?: { ... }              // 可选：MCP 元数据
}
```

## 工具注册（tools.ts）

### getAllBaseTools() — 静态数组注册

**不是**运行时 `register(name, handler)` 的插件注册表，而是编译期常量数组 + 条件展开：

```typescript
function getAllBaseTools(): Tools {
  return [
    // 文件操作
    FileReadTool,
    FileEditTool,
    FileWriteTool,

    // 搜索
    GlobTool,
    GrepTool,

    // 执行
    BashTool,
    ...(feature('NOTEBOOK_EDIT') ? [NotebookEditTool] : []),

    // Agent 系统
    AgentTool,
    ...(isTodoV2Enabled() ? [TaskCreateTool, TaskGetTool, ...] : []),

    // 条件加载（ant-only / 环境变量）
    ...(feature('REPL') ? [REPLTool] : []),
    ...(feature('WEB_SEARCH') ? [WebSearchTool, WebFetchTool] : []),
    // ...
  ]
}
```

### getTools() — 过滤后的工具列表

在 `getAllBaseTools()` 基础上做：
1. `isEnabled()` 过滤（禁用的工具移除）
2. deny 规则过滤（用户配置的 `--disallowed-tools`）
3. 特殊模式裁剪（Simple / REPL / Coordinator 模式）

### assembleToolPool() — 内置 + MCP 合并

```
内置工具（getTools 输出）
  + MCP 工具（运行时连接的 MCP 服务器提供）
  → 按名去重（内置优先）
  → 按名称排序（保证 prompt cache 稳定）
  → 最终工具池
```

## 工具实现模式

每个工具是一个目录，结构统一：

```
tools/FileEditTool/
  ├── FileEditTool.ts      ← buildTool() 定义
  ├── prompt.ts            ← 给模型的使用说明文本
  ├── constants.ts         ← 常量
  ├── utils.ts             ← 核心逻辑
  ├── UI.tsx               ← 终端 UI 渲染
  └── __tests__/           ← 测试
```

### 示例：FileEditTool

```typescript
export const FileEditTool = buildTool({
  name: 'Edit',
  searchHint: 'modify file contents in place',
  maxResultSizeChars: 100_000,
  strict: true,
  inputSchema: z.object({
    file_path: z.string(),
    old_string: z.string(),
    new_string: z.string(),
  }),

  async call(args, context) {
    // 读文件 → 查找 old_string → 替换为 new_string → 写回
    // 返回 { data: { success: true, diff: '...' } }
  },

  isReadOnly: () => false,
  isConcurrencySafe: () => false,
  isDestructive: () => false,

  async checkPermissions(input, context) {
    // 检查文件路径是否在允许范围内
  },
})
```

| 类别 | 工具名 | 说明 |

| **搜索** | `Glob`, `Grep`, `WebSearch`, `WebFetch` | 文件/网络搜索 |

| **远程** | `TerminalCapture`, `Tungsten`, `WebBrowser`, `ComputerUse` | 终端/浏览器/屏幕 |

## 工具执行引擎

### toolOrchestration.ts — 编排策略

```typescript
async function* runTools(toolUseBlocks, assistantMessages, canUseTool, context) {
  for (const { isConcurrencySafe, blocks } of partitionToolCalls(toolUseBlocks, context)) {
    if (isConcurrencySafe) {
      // 并行执行（最多 10 个）
      yield* runToolsConcurrently(blocks, ...)
    } else {
      // 串行执行
      yield* runToolsSerially(blocks, ...)
    }
  }
}
```

**分区规则**（`partitionToolCalls`）：

```
[Read, Grep, Grep]  → 并发安全 → 一批并行
[Bash(npm install)] → 非并发安全 → 一批串行
[Read, Read]        → 并发安全 → 一批并行
[Edit]              → 非并发安全 → 一批串行
```

- 连续的并发安全工具合并为一批并行
- 遇到非并发安全工具立即切分新批次
- 非并发安全工具永远独占一批

### toolExecution.ts — 单工具管线

完整执行管线：

```
① findToolByName()
   ↓ 未找到？→ 检查 deprecated alias 兜底
   ↓ 已 abort？→ 直接返回取消结果

② inputSchema.safeParse(input)
   ↓ 失败？→ 返回校验错误 + deferred schema 提示

③ validateInput(input, context)
   ↓ 失败？→ 返回校验错误

④ backfillObservableInput(input)
   ↓ 为观察者添加派生字段（不影响 API 原始输入）

⑤ runPreToolUseHooks(tool, input)
   ↓ 可修改 input / 拦截执行 / 附加消息

⑥ canUseTool(tool, input, context)
   ↓ 权限检查：auto-classifier + 规则匹配 + 用户对话框
   ↓ 拒绝？→ 返回权限拒绝

⑦ tool.call(args, context, canUseTool, msg, onProgress)
   ↓ ⭐ 执行工具核心逻辑
   ↓ 返回 ToolResult<T>

⑧ mapToolResultToToolResultBlockParam(result, toolUseId)
   ↓ 内部格式 → API tool_result 格式

⑨ 结果存储（超大结果持久化到磁盘）

⑩ runPostToolUseHooks(tool, input, result)
   ↓ 后处理钩子

⑪ yield UserMessage { tool_result }
```

### StreamingToolExecutor — 边流式边执行

常规模式下，模型输出完毕后才批量执行工具。StreamingToolExecutor 实现**边接收模型输出边执行**：

```
模型流式输出中...
  ├─ tool_use block 1 到达 → addTool() → 立即排队 → 开始执行
  ├─ tool_use block 2 到达 → addTool() → 排队（可能已开始并行）
  ├─ 模型继续输出 text...
  └─ tool_use block 3 到达 → addTool()

模型输出完毕 → getRemainingResults() → 等待未完成的工具
```

**并发控制**：
- 并发安全的工具立即开始执行，与模型输出和其他工具并行
- 非并发安全的工具阻塞队列扫描，保持顺序
- `siblingAbortController`：并行 Bash 等子进程出错时取消兄弟任务
- 用户中断时根据 `interruptBehavior` 决定 cancel 还是 block

## 权限系统（概述）

> 完整的权限系统分析见 → [07-权限系统.md](./07-权限系统.md)

### 三层权限检查

```
① 规则匹配（alwaysAllow / alwaysDeny / alwaysAsk）
   ↓ 来自用户配置（.claude/settings.json）
   ↓ 命中 → 直接决定

② Auto Classifier（自动分类器）
   ↓ 仅在 auto 模式下启用
   ↓ 分析工具调用是否安全

③ 用户对话框
   ↓ 非 auto 模式或分类器不确定时
   ↓ 弹出终端对话框让用户确认
```

### PermissionMode

| 模式 | 行为 |
|------|------|
| `default` | 每次都询问（除非有 always-allow 规则） |
| `auto` | 先走 auto-classifier，不确定才问 |
| `plan` | 只允许只读操作，写操作需确认 |
| `bypassPermissions` | 跳过所有检查（仅沙箱环境） |

## 设计亮点

| 亮点 | 说明 | 可借鉴场景 |
|------|------|----------|
| **buildTool + 默认值** | 统一工厂，保守默认（fail-closed），减少样板代码 | 任何需要大量插件/工具的系统 |
| **静态注册 + 条件编译** | `feature()` 在编译时消除未启用工具的代码 | 一套代码多版本发布 |
| **分区并发** | 自动将只读操作并行、写操作串行，无需工具自己处理 | 多工具/多任务的并发控制 |
| **流式执行** | 模型还在输出时就开始执行工具，节省总延迟 | AI Agent 的延迟优化 |
| **11 步管线** | 查找→校验→钩子→权限→执行→格式化→存储→后处理，每步可扩展 | 需要安全/审计/扩展的执行管线 |
| **Zod schema** | 编译时类型安全 + 运行时校验，一份 schema 两用 | 类型驱动的 API 设计 |
