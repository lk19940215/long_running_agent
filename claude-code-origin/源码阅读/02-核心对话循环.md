# 02 - 核心对话循环

> 一句话：用户输入 → 消息预处理 → 上下文压缩 → 调 API → 流式接收 → 执行工具 → 拼接结果 → 循环或结束。

## 为什么这是"灵魂"？

启动流程（01）把所有配置装配好，最终目标是调 `launchRepl()` 启动交互界面。但用户输入之后发生了什么？答案就在 `QueryEngine.ts` 和 `query.ts` —— 它们构成了 Claude Code 的**核心对话循环**，驱动 LLM 多轮对话和工具调用。

如果 `main.tsx` 是"组装车间"，那么 `QueryEngine` + `query` 就是"发动机"。

## 全景图

有两条路径进入 query.ts，REPL（CLI 交互）路径和 SDK（程序调用）路径：

### REPL 路径（CLI 交互，主路径）

```
用户按回车（详见 01-启动流程.md 的 6 层管道）
  ↓
REPL.tsx: onSubmit → handlePromptSubmit → processUserInput → onQuery
  ↓
onQueryImpl()                          ← 组装上下文 + 直接调 query()
  ├─ 并行加载（Promise.all）：
  │   ├─ getSystemPrompt(tools, model, dirs, mcpClients)  — 基础提示词
  │   ├─ getUserContext()                                  — 用户上下文
  │   └─ getSystemContext()                                — 系统上下文
  ├─ buildEffectiveSystemPrompt() — 合并自定义/追加提示词
  ↓
  for await (const event of query({    ← 直接调 query()
    messages, systemPrompt,
    userContext, systemContext,
    canUseTool, toolUseContext
  })) {
    onQueryEvent(event)                ← 流式事件 → React state
  }
  ↓
query() → queryLoop()                 ← 请求级：核心 while(true) 循环
  ↓   ↓
  ↓   ① 上下文压缩管道                ← snip → microcompact → collapse → autocompact
  ↓   ② 调 Claude API                 ← 流式接收，边收边执行工具
  ↓   ③ 工具执行                      ← 并行/串行执行，收集 tool_result
  ↓   ④ 错误恢复                      ← prompt-too-long / max-output-tokens / fallback
  ↓   ⑤ 停止钩子                      ← 检查是否需要阻止或追加
  ↓   ⑥ 附件注入                      ← 内存/技能发现/排队命令
  ↓   ⑦ 状态转移                      ← 拼接消息 → state = next → continue
  ↓
  (无 tool_use → 退出 while)
  return { reason: 'completed' }
  ↓
onQueryEvent()                         ← "搬运工"，不做业务决策
  ├─ compact 边界 → 替换全部消息
  ├─ ephemeral progress → 替换最后一条
  └─ 其他 → append 到 messages
  ↓
REPL 展示结果，等待下一轮输入
```

### SDK 路径（程序调用）

```
SDK 调用方
  ↓
QueryEngine.submitMessage()            ← 会话级：管理消息历史、使用量、文件缓存
  ↓ processUserInput()                 ← 处理斜杠命令、附件、模型切换
  ↓ fetchSystemPromptParts()           ← 组装系统提示词
  ↓ for await (query({...}))           ← 调用核心循环
  ↓ yield SDK 消息（助手/用户/系统/流事件）
  ↓ yield { type: 'result', ... }      ← 告知调用方：成功/错误/超限
```

### systemPrompt 组装过程

onQueryImpl 中的提示词组装分三步：

```
① getSystemPrompt(tools, model, dirs, mcpClients)
   → 基础提示词：Claude 的角色定义 + 每个工具的 prompt() 描述 + MCP 工具描述

② getUserContext() + getCoordinatorUserContext()
   → 用户上下文：cwd、OS、git 状态、MCP 服务器信息、scratchpad、终端焦点状态
   → 注入方式：prependUserContext() 作为第一条 user 消息前缀（不在 system prompt 中）

③ buildEffectiveSystemPrompt({
     mainThreadAgentDefinition,    ← agent 定义前缀（如有）
     customSystemPrompt,           ← 用户 rules（.claude/rules 等）
     defaultSystemPrompt,          ← ① 的结果
     appendSystemPrompt,           ← 追加提示词
   })
   → 最终 systemPrompt = agent定义 + 默认 + 自定义 + 追加

④ getSystemContext()
   → 系统上下文（环境信息）
   → 注入方式：appendSystemContext() 追加到 systemPrompt 末尾
```

## 两个核心文件的分工

| 文件 | 层级 | 职责 | 生命周期 |
|------|------|------|---------|
| `QueryEngine.ts` | 会话级 | 管理整个对话的消息历史、权限拒绝记录、累计使用量、文件状态缓存 | 一个对话 = 一个 QueryEngine 实例 |
| `query.ts` | 请求级 | 一次用户输入后的完整处理循环：压缩 → API → 工具 → 循环 | 一次 submitMessage = 一次 query() 调用 |

**关系**：`QueryEngine.submitMessage()` 调用 `query()`，`query()` 是一个 async generator，每 yield 一条消息，`QueryEngine` 就处理一条（记录转录、追踪使用量、推送给 SDK/UI）。

## QueryEngine.ts 详解

### 类结构

```typescript
class QueryEngine {
  // 核心状态
  private mutableMessages: Message[]         // 消息历史（整个对话累积）
  private abortController: AbortController   // 中断控制
  private permissionDenials: SDKPermissionDenial[]  // 权限拒绝记录
  private totalUsage: NonNullableUsage       // 累计 token 使用量
  private readFileState: FileStateCache      // 文件读取缓存
  private discoveredSkillNames: Set<string>  // 技能发现追踪

  // 核心方法
  async *submitMessage(prompt, options)      // ⭐ 提交一轮对话
  interrupt()                                // 中断当前请求
  getMessages()                              // 获取消息历史
  setModel(model)                            // 切换模型
}
```

### submitMessage() 生命周期

```
submitMessage(prompt)
  ↓
  ① 构建上下文
     - setCwd()                           设置工作目录
     - fetchSystemPromptParts()           组装系统提示词 + 用户上下文
     - 注册结构化输出钩子（如有 jsonSchema）
     - 构建 processUserInputContext       工具/命令/状态的上下文对象
  ↓
  ② 处理用户输入
     - processUserInput()                 处理斜杠命令（/compact, /model 等）
     - mutableMessages.push(...)          用户消息入队
     - recordTranscript()                 持久化到转录文件
  ↓
  ③ 调用核心循环
     - for await (const message of query({...}))
     - 逐条处理 yield 出来的消息：
       - assistant → 推入 mutableMessages，yield 给 SDK
       - user      → 推入 mutableMessages，轮次计数 +1
       - system    → 处理压缩边界、API 重试、snip 回放
       - stream_event → 累计 token 使用量
       - attachment → 处理 max_turns / structured_output / 排队命令
       - tool_use_summary → 透传给 SDK
  ↓
  ④ 结果收尾
     - 检查预算超限（maxBudgetUsd）
     - 检查结构化输出重试超限
     - 提取最终文本结果
     - yield { type: 'result', subtype: 'success' | 'error_*' }
```

### ask() 便捷包装

`ask()` 是 `QueryEngine` 的简化包装，用于一次性调用场景（SDK / print 模式）：

```typescript
async function* ask({ prompt, tools, ... }) {
  const engine = new QueryEngine({ ... })
  yield* engine.submitMessage(prompt)
}
```

### State — 循环状态机

每次循环迭代都从 `state` 解构出当前状态，循环末尾用 `state = next` 更新。
State 是理解整个 `queryLoop()` 的钥匙——**所有 continue 站点都通过构造新 State 来驱动下一轮**。

```typescript
type State = {
  messages: Message[]                    // 当前消息序列（会被压缩管道修改）
  toolUseContext: ToolUseContext          // 工具执行上下文（工具列表、MCP、权限等）
  autoCompactTracking: ...               // 自动压缩追踪（turnId/turnCounter/consecutiveFailures）
  maxOutputTokensRecoveryCount: number   // 输出截断恢复次数（最多 3 次）
  hasAttemptedReactiveCompact: boolean   // 是否已尝试响应式压缩（防止死循环）
  maxOutputTokensOverride: number        // 输出 token 限制覆盖（8k→64k 提升）
  pendingToolUseSummary: Promise<...>    // 上一轮工具摘要（Haiku 异步生成，下一轮 yield）
  stopHookActive: boolean               // 停止钩子是否激活（阻止本轮退出）
  turnCount: number                     // 当前轮次（从 1 开始）
  transition: Continue | undefined      // 本次循环的转移原因（undefined = 首次迭代）
}
```

**转移原因（transition.reason）** 揭示了为什么循环会 `continue`：

| reason | 含义 | 所在子块 |
|--------|------|---------|
| `next_turn` | 正常：工具执行完毕，继续下一轮 | 🅽 |
| `collapse_drain_retry` | 上下文折叠排空后重试（413 恢复路径 1） | 🅻 |
| `reactive_compact_retry` | 响应式压缩后重试（413 恢复路径 2） | 🅻 |
| `max_output_tokens_escalate` | 输出 token 限制从 8k 提升到 64k 后重试 | 🅻 |
| `max_output_tokens_recovery` | 输出截断恢复中（注入 meta 消息让模型续写，最多 3 次） | 🅻 |
| `stop_hook_blocking` | 停止钩子阻止后重试 | 🅻 |
| `token_budget_continuation` | Token 预算续行（预算未耗尽，继续执行） | 🅻 |

**8 个 continue 站点**（每个都构造新 State，全部用 `state = next; continue` 模式）：

```
🅶 autocompact 成功后 — tracking 重置（非 continue，但修改了 state 的 tracking）
🅻 collapse_drain_retry — 排空折叠后重试
🅻 reactive_compact_retry — 响应式压缩后重试
🅻 max_output_tokens_escalate — 8k→64k 提升
🅻 max_output_tokens_recovery — 注入恢复消息续写
🅻 stop_hook_blocking — 注入 hook 错误消息重试
🅻 token_budget_continuation — 预算续行
🅽 next_turn — 正常路径，工具结果 + 附件拼接完毕
```

### ① 上下文压缩管道

每次循环开头，消息序列要经过 **4 层压缩**，顺序不能乱：

```
原始消息
  ↓ applyToolResultBudget()     0. 工具结果预算：超大结果持久化到磁盘
  ↓ snipCompactIfNeeded()       1. Snip：按标记裁剪历史（旧消息变占位符）
  ↓ microcompact()              2. 微压缩：移除过时的工具调用详情
  ↓ applyCollapsesIfNeeded()    3. 上下文折叠：将已处理的段落折叠为摘要
  ↓ autocompact()               4. 自动压缩：超过阈值时全量压缩为摘要
  ↓
压缩后的消息 → 送给 API
```

**为什么顺序重要？**
- Snip → Microcompact：先裁剪旧消息，再压缩工具详情
- Collapse 在 Autocompact 之前：如果折叠就够了，就不需要全量压缩（保留更细粒度的上下文）
- Autocompact 是"最后手段"：只在前面都不够时触发

**阈值判断**：
```
有效上下文窗口 = 模型上下文窗口 - 预留输出 token（20000）
自动压缩阈值 = 有效上下文窗口 - 13000（缓冲区）
阻塞限制阈值 = 更高的门槛（用于手动 /compact 的保留空间）
```

### ② 调 Claude API（流式）

```
messagesForQuery
  ↓ prependUserContext()        前置用户上下文
  ↓ appendSystemContext()       追加系统上下文
  ↓ deps.callModel({            调用模型 API
  ↓   messages, systemPrompt,
  ↓   thinkingConfig, tools,
  ↓   signal, options: {
  ↓     model, fastMode,
  ↓     fallbackModel,
  ↓     taskBudget, ...
  ↓   }
  ↓ })
  ↓
  for await (const message of stream) {
    // 流式接收每个 content block
    - text block → yield 给外层
    - tool_use block → 收集到 toolUseBlocks[]
    - thinking block → yield（可能被扣留）
    
    // 扣留机制：可恢复的错误不立即 yield
    - prompt-too-long → 扣留，等后面的恢复逻辑
    - max-output-tokens → 扣留，等后面的恢复逻辑
    - media-size-error → 扣留，等后面的恢复逻辑
    
    // 流式工具执行（边收边跑）
    if (streamingToolExecutor) {
      streamingToolExecutor.addTool(toolBlock)
      for (result of getCompletedResults()) yield result
    }
  }
```

**模型回退**：如果主模型不可用，自动切换到 fallbackModel 重试，清空之前的部分结果。

### ③ 工具执行

```
toolUseBlocks[] (模型返回的工具调用请求)
  ↓
  两种执行模式：
  
  A. 流式执行（StreamingToolExecutor）
     边接收模型输出边执行工具
     → 模型输出 tool_use → 立即开始执行 → 无需等模型输出完
     → 大幅减少总延迟（API 5-30s 期间工具已在跑）
  
  B. 传统执行（runTools）
     等模型输出完毕后批量执行
     → partitionToolCalls() 分区：
       - 并发安全（只读）→ 最多 10 个并行
       - 非并发安全（写操作）→ 串行执行
```

**工具执行流程**（每个工具）：

```
toolUseBlock { name, input, id }
  ↓ findToolByName()            查找工具定义
  ↓ tool.validateInput()        输入校验
  ↓ canUseTool()                权限检查（may prompt user）
  ↓ tool.call(args, context)    ⭐ 执行工具逻辑
  ↓ tool.mapToolResultToToolResultBlockParam()  格式化结果
  ↓
yield UserMessage { tool_result: { tool_use_id, content } }
```

### ④ 错误恢复

query.ts 有一套精密的错误恢复机制，处理各种 API 错误：

| 错误类型 | 恢复策略 | 最大重试 |
|---------|---------|---------|
| **prompt-too-long (413)** | 先尝试 Context Collapse 排空 → 再尝试 Reactive Compact 全量压缩 | 各 1 次 |
| **max-output-tokens** | 先尝试 8k→64k 提升限制 → 再注入 meta 消息让模型续写 | 提升 1 次 + 续写 3 次 |
| **media-size-error** | Reactive Compact 剥离并重试 | 1 次 |
| **模型不可用** | FallbackTriggeredError → 切换备用模型重试 | 1 次 |
| **其他 API 错误** | 表面化错误、记录日志 | 0 次 |

**扣留（withhold）机制**：可恢复的错误不立即 yield 给外层，而是扣留在 `assistantMessages[]` 中。如果恢复成功，扣留的错误消息被丢弃；如果恢复失败，才 yield 出去。

---

### ⑤ 停止钩子（Stop Hooks）

当模型返回且**没有** tool_use 时（`needsFollowUp = false`），在最终返回前检查停止钩子：

```
handleStopHooks()
  ↓ 检查是否有阻止条件（blocking errors）
  ↓ 检查是否需要阻止继续（preventContinuation）
  ↓
  if (blockingErrors) → 注入错误消息 → state = next → continue（再跑一轮）
  if (preventContinuation) → return { reason: 'stop_hook_prevented' }
  otherwise → 正常退出
```

---

### ⑥ 附件注入

工具执行完毕后，在组装下一轮消息前，注入额外的上下文：

```
getAttachmentMessages()          获取附件消息（文件变更追踪等）
filterDuplicateMemoryAttachments() 内存预取结果（去重）
collectSkillDiscoveryPrefetch()  技能发现结果
getCommandsByMaxPriority()       排队命令（任务通知等）
```

这些附件作为 `tool_result` 追加到消息序列中，让模型在下一轮看到。

### ⑦ 状态转移 — 循环还是结束？

```
                        有 tool_use？
                       /           \
                     是              否
                     ↓               ↓
              拼接消息：          错误恢复逻辑
              messages +         ↓
              assistantMessages + 停止钩子检查
              toolResults        ↓
                     ↓           return { reason }
              检查 maxTurns
                     ↓

                messages: [...新消息],
                turnCount: turnCount + 1,
                transition: { reason: 'next_turn' },
                ...
              }

```

## Tool.ts — 工具类型定义

### Tool 核心类型

每个工具都是一个满足 `Tool` 接口的对象：

```typescript
type Tool = {
  name: string                           // 工具名称
  aliases?: string[]                     // 别名（重命名后的兼容）
  inputSchema: ZodSchema                 // 输入 schema（Zod v4）
  
  // 核心方法
  call(args, context, canUseTool, parentMessage, onProgress)
    → Promise<ToolResult>               // ⭐ 执行工具逻辑
  prompt(options) → Promise<string>      // 生成给模型的工具说明
  description(input, options) → Promise<string>  // 工具描述
  
  // 权限
  checkPermissions(input, context)       // 权限检查

}
```

## 上下文压缩策略汇总

| 策略 | 触发条件 | 原理 | Feature Gate |
|------|---------|------|-------------|
| **Snip** | 消息带有 snip 标记 | 按标记裁剪旧消息为占位符 | `HISTORY_SNIP` |
| **Microcompact** | 每轮自动 | 移除过时工具调用的详细输入/输出 | 默认启用 |
| **Context Collapse** | 段落可折叠 | 将已完成的段落折叠为摘要（保留细节） | `CONTEXT_COLLAPSE` |
| **Autocompact** | token > 阈值 | 全量压缩为一段摘要 + 保留尾部消息 | 默认启用 |
| **Reactive Compact** | API 返回 413 | 紧急压缩，剥离超大内容后重试 | `REACTIVE_COMPACT` |

**优先级**：Snip → Microcompact → Collapse → Autocompact → Reactive（最后手段）

## 设计亮点

| 亮点 | 说明 | 可借鉴场景 |
|------|------|----------|
| **Async Generator 管道** | `query()` 用 `yield*` 驱动，每个消息即时推送，不缓冲 | 需要流式处理的长任务 |
| **扣留-恢复模式** | 可恢复错误扣留不 yield，恢复成功丢弃、失败才暴露 | 任何有自动重试的流式系统 |
| **流式工具执行** | 模型输出和工具执行并行（StreamingToolExecutor） | AI Agent 工具延迟优化 |
| **工具编排分区** | 只读工具并行、写操作串行，自动分区 | 需要并发执行但要保证顺序的系统 |
| **5 层压缩管道** | 从"便宜的裁剪"到"昂贵的全量压缩"，逐层升级 | 长对话/上下文管理 |
| **状态机驱动** | State + transition.reason 让循环逻辑清晰可测 | 复杂状态流转 |
