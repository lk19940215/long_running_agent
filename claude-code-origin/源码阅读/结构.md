# Claude Code 源码架构总览

> 本仓库是 Claude Code v2.1.88 源码泄露后的社区还原版。
> Anthropic 2026年3月在 npm 发布时意外包含了 source map，暴露了 50万+ 行源码。
> Claude Code 并未官方开源，Anthropic 已发起 DMCA 投诉。
[模拟器](https://www.xuanyuancode.com/learn-claude-code/simulator)

## 技术栈

| 层面 | 技术 |
|------|------|
| 运行时 | Bun 1.3.5+（构建/运行/包管理） |
| 语言 | TypeScript (ESM, `.ts` / `.tsx`) |
| UI 框架 | React + 自定义 Ink 终端渲染器 |
| 状态管理 | 轻量 subscribe/getState/setState（非 Redux） |
| Schema 校验 | Zod v4 |
| API SDK | `@anthropic-ai/sdk` |
| 构建宏 | `bun:bundle` 的 `feature()` 条件编译 |
| 插件协议 | MCP (Model Context Protocol) |
| CLI 框架 | Commander.js |

## 阅读深度建议

> 读源码不是为了复刻实现，而是为了提炼**可迁移的设计模式**。

| 层次 | 目标 | 方法 | 投入 |
|------|------|------|------|
| **L1 架构层** | 理解设计模式的**为什么** | 读本文档 + 各章节头注释 + 核心类型定义 | **80% 时间** |
| **L2 机制层** | 理解某个具体机制的**怎么做** | 打开对应 `#region` 块，读骨架逻辑 | 按需 15% |
| **L3 实现层** | 理解每行代码的**细节** | 逐行阅读 | 仅改代码时 5% |

**建议**：如果你不打算改 Claude Code 的代码，到 L1 + 部分 L2 就够了。把省下的时间用来设计你自己的 AI 工具架构。

## 可迁移设计模式 — 从 Claude Code 带走什么

以下 6 个模式是 Claude Code 最值得借鉴的架构思想，按在源码中的位置标注了出处。

### 模式 1：AsyncGenerator 管道
**出处**：`query.ts` → `REPL.tsx` 的 `for-await` 消费链

**问题**：AI 对话是长时间运行的流式过程，需要在产生结果的同时推给 UI。

**解法**：`query()` → `queryLoop()` 都是 `async function*`，用 `yield*` 驱动：

```
queryLoop() yield → query() yield* → REPL onQueryImpl for-await → onQueryEvent → React state
```

**核心优势**：
- **零缓冲**：每个事件立即推给消费方，不需要 callback 或 EventEmitter
- **背压自然**：消费方没 pull，生产方自动暂停（generator 协议）
- **组合性**：`yield*` 透传不丢事件，多层嵌套不增加复杂度
- **资源清理**：generator 的 `return()` / `throw()` 让取消和清理路径一致

### 模式 2：状态机驱动循环
**出处**：`query.ts` `#region 🅵` 的 `State` 类型和 7 个 continue 站点

**问题**：while(true) 循环有多个 continue 路径（正常下一轮、错误恢复、预算续行等），逻辑容易混乱。

**解法**：`State` 对象 + `transition.reason`：

```typescript
state = { messages, toolUseContext, transition: { reason: 'next_turn' }, ... }
continue
```

**核心优势**：
- **可测试**：测试只需断言 `transition.reason` 而非检查消息内容
- **可追踪**：日志记录 `transition.reason` 就知道循环在干什么
- **无隐式状态**：所有状态集中在一个对象，不会遗漏某个变量的更新

### 模式 3：分层压缩管道
**出处**：`query.ts` `#region 🅶` + `services/compact/` 目录

**问题**：长对话会超出上下文窗口，需要压缩历史，但全量压缩太贵。

**解法**：5 层管道，从便宜到昂贵逐层升级：

```
Snip（裁剪标记）→ Microcompact（移除过时细节）→ Collapse（段落折叠）→ Autocompact（全量摘要）→ Reactive（紧急 413 恢复）
```

**设计原则**：最小干预、顺序依赖、独立开关、阈值管理。
**关键启示**：不要一步到位，设计多层渐进式策略。

### 模式 4：扣留-恢复（Withhold-Recover）
**出处**：`query.ts` `#region 🅸` 的 `withheld` 逻辑 + `#region 🅻` 的恢复分支

**问题**：流式 API 中遇到可恢复错误，如果立即暴露，调用方可能终止会话。

**解法**：

```
API 返回 413 → 扣留（不 yield）→ 尝试恢复 → 成功丢弃 / 失败才暴露
```

**核心优势**：调用方看到的要么是成功，要么是最终不可恢复错误——没有中间状态泄漏。

### 模式 5：流式工具执行
**出处**：`services/tools/StreamingToolExecutor.ts`

**问题**：Claude API 流式回复 5-30 秒，等回复完再执行工具，白白浪费时间。

**解法**：API 流式输出期间，一旦收到 `tool_use` block 就立即开始执行，**节省 30-50% 延迟**。

### 模式 6：工具编排分区
**出处**：`services/tools/toolOrchestration.ts` 的 `partitionToolCalls()`

**问题**：一次回复可能调用多个工具，全串行太慢，全并行可能冲突。

**解法**：`isConcurrencySafe()` 自动分区——只读工具并行（最多 10 个），写操作串行。
