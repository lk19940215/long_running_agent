# 06 - 系统提示词构建

> **一句话总结**：Claude Code 的系统提示词不是一个静态字符串，而是一套**分层组装、缓存友好**的动态拼接系统——静态段全局缓存、动态段按回合刷新、用户上下文按会话缓存。

---

## 为什么重要？

系统提示词决定了 Claude 的"人格"、"能力边界"和"行为规则"。理解这套拼接机制，你就理解了：
- 为什么 Claude Code 知道当前工作目录、Git 状态、操作系统信息
- CLAUDE.md 是怎么注入到 AI 上下文的
- MCP 工具说明是如何随连接状态动态变化的
- 为什么有些提示词改变不会破坏 prompt cache

## 逐层详解

### 第一层：systemPrompt —— 主提示词正文

**入口函数**：`getSystemPrompt()` in `constants/prompts.ts`

#### 静态段（DYNAMIC_BOUNDARY 之前）

这些段落跨组织/跨用户完全相同，可以用全局缓存 `scope: 'global'`：

| 段落函数 | 内容概要 |
|---------|---------|
| `getSimpleIntroSection()` | 角色定义："你是一个帮助用户完成软件工程任务的交互式智能体" + 安全指令 |
| `getSimpleSystemSection()` | 系统规则：Markdown 格式、权限模式、`<system-reminder>` 标签说明、Hooks |
| `getSimpleDoingTasksSection()` | 任务指南：代码风格（不过度设计、不添加多余注释）、安全编码、错误诊断策略 |
| `getActionsSection()` | 操作准则：可逆/不可逆操作的确认机制，"三思而后行" |
| `getUsingYourToolsSection()` | 工具使用：优先用专用工具而非 Bash，并行调用策略 |
| `getSimpleToneAndStyleSection()` | 语气：不用 emoji、引用代码用 `file:line` 格式 |
| `getOutputEfficiencySection()` | 输出效率：简洁直接，先说结论再说原因 |

#### DYNAMIC_BOUNDARY 分界线

```typescript
export const SYSTEM_PROMPT_DYNAMIC_BOUNDARY = '__SYSTEM_PROMPT_DYNAMIC_BOUNDARY__'
```

这个标记的作用是**分割缓存域**：
- 标记之前的内容 → `scope: 'global'`，所有用户共享缓存
- 标记之后的内容 → 按用户/会话变化，不参与全局缓存

#### 动态段（DYNAMIC_BOUNDARY 之后）

通过 `systemPromptSection()` / `DANGEROUS_uncachedSystemPromptSection()` 注册：

```typescript
// 普通段：首次计算后缓存，/clear 或 /compact 时清除
systemPromptSection('session_guidance', () => getSessionSpecificGuidanceSection(...))

// 危险段：每回合重新计算，会破坏 prompt cache
DANGEROUS_uncachedSystemPromptSection('mcp_instructions', () => getMcpInstructionsSection(...),
  'MCP servers connect/disconnect between turns'  // 必须说明原因
)
```

关键动态段：

| 段落 | 缓存策略 | 说明 |
|------|---------|------|
| `session_guidance` | 缓存 | Agent 工具使用、探索代理、Skill 搜索、验证代理等 |
| `memory` | 缓存 | `loadMemoryPrompt()` —— memdir 系统的持久记忆 |
| `env_info_simple` | 缓存 | 工作目录、Git、平台、Shell、模型名称、知识截止日期 |
| `language` | 缓存 | 用户语言偏好（如 "Always respond in 中文"） |
| `mcp_instructions` | **每回合** | MCP 服务器的使用说明，因为服务器可能随时连接/断开 |
| `scratchpad` | 缓存 | 临时文件目录说明 |
| `frc` | 缓存 | 工具结果自动清理说明（配合 cached microcompact） |

### 第二层：userContext —— 用户上下文

**入口函数**：`getUserContext()` in `context.ts`

```typescript
export const getUserContext = memoize(async () => {
  // 1. 加载 CLAUDE.md 文件（多级：Managed → User → Project → Local）
  const claudeMd = getClaudeMds(filterInjectedMemoryFiles(await getMemoryFiles()))
  
  // 2. 缓存供 yoloClassifier 使用（避免循环依赖）
  setCachedClaudeMdContent(claudeMd || null)
  
  return {
    ...(claudeMd && { claudeMd }),
    currentDate: `Today's date is ${getLocalISODate()}.`,
  }
})
```

**CLAUDE.md 加载顺序**（`getMemoryFiles()`）：

```
1. Managed（策略设置）     → ~/.claude/CLAUDE.md（组织管理）
2. User（用户设置）        → ~/.claude/CLAUDE.md + ~/.claude/rules/*.md
3. Project（项目设置）     → 从 CWD 向上遍历：CLAUDE.md + .claude/CLAUDE.md + .claude/rules/*.md
4. Local（本地设置）       → CLAUDE.local.md（gitignore 的）
```

> 这也是为什么 `CLAUDE.md` 放在项目根目录就能生效的原因。

### 第三层：systemContext —— 系统上下文

**入口函数**：`getSystemContext()` in `context.ts`

```typescript
export const getSystemContext = memoize(async () => {
  // 并行获取 Git 信息
  const gitStatus = await getGitStatus()
  // Git 信息包括：分支、主分支、用户名、状态（截断到 2000 字符）、最近 5 条 commit
  
  return {
    ...(gitStatus && { gitStatus }),
    // 可选的 cache breaker（仅内部调试用）
  }
})
```

## 优先级路由：buildEffectiveSystemPrompt

当多种提示词来源并存时，`utils/systemPrompt.ts` 的 `buildEffectiveSystemPrompt()` 决定最终使用哪个：

```
优先级（从高到低）：

0. overrideSystemPrompt    → 完全替代所有其他提示词（loop mode 等场景）
1. Coordinator prompt      → Coordinator 模式时使用专用提示词
2. Agent prompt            → 主线程 Agent 定义的提示词
   - Proactive 模式：Agent 提示词 追加到 default 之后
   - 普通模式：Agent 提示词 替换 default
3. customSystemPrompt      → --system-prompt 命令行参数
4. defaultSystemPrompt     → getSystemPrompt() 的标准输出

+ appendSystemPrompt 始终追加到末尾（除 override 外）
```

## 组装时序：从 QueryEngine 到 API

```
QueryEngine.ask()
  │
  ├── fetchSystemPromptParts()        // 并行获取三部分
  │   ├── getSystemPrompt(tools, model, dirs, mcp)   → defaultSystemPrompt[]
  │   ├── getUserContext()                             → { claudeMd, currentDate }
  │   └── getSystemContext()                           → { gitStatus }
  │
  ├── buildEffectiveSystemPrompt()    // 优先级路由
  │   → systemPrompt: SystemPrompt
  │
  └── query(systemPrompt, userContext, systemContext, ...)
      │
      └── API 请求组装
          system: [
            ...systemPrompt,         // 主提示词（多个字符串段）
            userContext.claudeMd,     // CLAUDE.md
            userContext.currentDate,  // 日期
            systemContext.gitStatus,  // Git 快照
          ]
```

## 设计亮点

### 1. 双级缓存策略

- **systemPromptSection**：首次计算后 memoize，`/clear` 或 `/compact` 清除
- **DANGEROUS_uncachedSystemPromptSection**：每回合重算，仅用于确实需要实时性的内容（MCP）

这避免了"所有动态内容都破坏缓存"的浪费。

### 2. DYNAMIC_BOUNDARY 分割

静态段所有用户共享同一份 prompt cache（Blake2b 前缀哈希），极大提升缓存命中率。
动态段的变化不影响静态段的缓存。

### 3. 三部分并行获取

`fetchSystemPromptParts()` 用 `Promise.all` 同时获取 systemPrompt、userContext、systemContext，
避免串行等待（尤其 `getGitStatus()` 需要执行 git 命令）。

### 4. CLAUDE.md 的四级优先级

Managed > User > Project > Local，既支持组织级策略控制，又允许项目级和个人级定制。
`CLAUDE.local.md` 被 gitignore，适合放个人偏好。

### 5. 安全指令硬编码

`CYBER_RISK_INSTRUCTION` 由 Safeguards 团队维护，硬编码在所有提示词路径中：
```
"Assist with authorized security testing... Refuse requests for destructive techniques..."
```
确保无论哪种提示词路由，安全边界始终存在。
