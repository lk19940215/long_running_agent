# 07 - 权限系统

> **一句话总结**：Claude Code 的权限系统是一个**三层判决链**——工具自身的 `checkPermissions` → 全局规则匹配 → 模式变换（dontAsk/auto/bypassPermissions），最终产生 `allow`、`deny` 或 `ask` 三种决策。

---

## 为什么重要？

权限系统是 Claude Code 的**安全底座**。每一次工具调用都必须经过权限检查：
- 它决定了哪些操作自动执行、哪些需要用户确认、哪些被直接拒绝
- 它是 Auto 模式（AI 自主分类器）和安全检查的枢纽
- 它连接了 UI 层（权限弹窗）和核心层（工具执行）

---

## 全景图：权限判决链

```
工具调用请求
  │
  ▼
hasPermissionsToUseToolInner()          ← 核心判决函数
  │
  ├── 1a. 整个工具被 deny？              → deny
  ├── 1b. 整个工具被 ask？               → ask（除非 sandbox 自动放行）
  ├── 1c. tool.checkPermissions()        → 工具自身判断
  │   ├── 1d. 工具返回 deny？            → deny
  │   ├── 1e. requiresUserInteraction？  → ask（即使 bypass 模式）
  │   ├── 1f. 内容级 ask 规则？          → ask（即使 bypass 模式）
  │   └── 1g. safetyCheck？              → ask（bypass 免疫）
  ├── 2a. bypassPermissions 模式？       → allow
  ├── 2b. 整个工具 alwaysAllow？         → allow
  └── 3.  passthrough → ask             → 默认需要用户确认
  │
  ▼
hasPermissionsToUseTool()               ← 外层包装
  │
  ├── allow → 重置连续拒绝计数
  ├── ask + dontAsk 模式？              → deny
  ├── ask + auto 模式？                 → AI 分类器决策
  │   ├── safetyCheck(非分类器可批) → deny/ask
  │   ├── 安全工具快速允许             → allow
  │   ├── acceptEdits 快速允许         → allow
  │   └── yoloClassifier 判断          → allow/deny
  └── ask + shouldAvoidPermissionPrompts？ → deny（headless 场景）
  │
  ▼
useCanUseTool() Hook                    ← REPL 层
  │
  ├── allow → 直接通过
  ├── deny  → 通知用户 + 返回拒绝
  └── ask   → 根据场景分发：
      ├── coordinatorHandler   → 协调者模式处理
      ├── swarmWorkerHandler   → Swarm 工作者处理
      └── interactiveHandler   → 交互式弹窗 → 用户决策
```

## 逐层详解

### 第一层：工具自身判断 (tool.checkPermissions)

每个工具实现自己的 `checkPermissions()`，返回 `PermissionResult`：

```typescript
type PermissionResult = {
  behavior: 'allow' | 'deny' | 'ask' | 'passthrough'
  message?: string
  decisionReason?: PermissionDecisionReason
  updatedInput?: Input  // 可以修改工具输入
  suggestions?: PermissionSuggestion[]
}
```

**`passthrough`** 是特殊状态：表示"工具自己不判断，交给全局规则"。`buildTool` 的默认实现就是 `{ behavior: 'allow' }`。

典型工具实现：

| 工具 | checkPermissions 策略 |
|------|---------------------|
| `BashTool` | 检查命令前缀规则、Sandbox 状态、敏感路径 |
| `FileEditTool` | 检查文件路径安全性（.git/、.claude/、shell 配置文件） |
| `FileReadTool` | 通常 passthrough，但检查路径安全性 |
| `MCPTool` | 始终 passthrough，需要全局规则判断 |

### 第二层：全局规则匹配

`hasPermissionsToUseToolInner()` 的判决步骤：

```
步骤 1：否定检查（防御优先）
  1a. getDenyRuleForTool()     → 工具级 deny 规则（如 Bash(rm -rf:*) deny）
  1b. getAskRuleForTool()      → 工具级 ask 规则
  1c. tool.checkPermissions()  → 工具自身判断
  1d-1g. 细分处理工具返回的 deny/ask/safetyCheck

步骤 2：肯定检查
  2a. bypassPermissions 模式   → 全部允许（除了 safetyCheck）
  2b. toolAlwaysAllowedRule()  → 工具在 alwaysAllow 列表中

步骤 3：兜底
  passthrough → 转为 ask → 需要用户确认
```

### 第三层：模式变换

`hasPermissionsToUseTool()`（外层）对 `ask` 决策进行模式变换：

| 权限模式 | 对 ask 的处理 |
|---------|-------------|
| `default` | 保持 ask → 弹窗询问用户 |
| `dontAsk` | ask → deny（不打扰用户，直接拒绝） |
| `bypassPermissions` | 在内层已处理为 allow |
| `auto` | ask → AI 分类器判断 → allow 或 deny |
| `plan` | 如果 bypass 可用则 allow，否则 ask |

### Auto 模式的 AI 分类器

当权限模式为 `auto` 时，`ask` 决策会走 AI 分类器：

```
ask + auto 模式
  │
  ├── safetyCheck（不可分类）？  → deny 或保持 ask
  ├── requiresUserInteraction？  → 保持 ask
  ├── PowerShell 且未启用？     → deny 或保持 ask
  │
  ├── 连续拒绝过多？             → 回退到弹窗 ask
  │
  ├── 安全工具快速路径？          → allow
  │   (FileReadTool, GlobTool, GrepTool 等只读工具)
  │
  ├── acceptEdits 快速路径？     → allow
  │   (BashTool 的 accept_edits, FileEditTool 带 safety 过的)
  │
  └── yoloClassifier()          → AI 分析命令安全性
      ├── 安全 → allow
      └── 不安全 → deny + 通知用户
```

## 权限规则来源

规则可以来自多个层级，合并后生效：

```
┌─ policySettings   ← 组织级策略（只读）
├─ flagSettings     ← Feature flag（只读）
├─ userSettings     ← ~/.claude/settings.json
├─ projectSettings  ← .claude/settings.json
├─ localSettings    ← .claude/settings.local.json
├─ cliArg           ← 命令行参数
├─ command          ← 命令触发
└─ session          ← 会话中用户授权
```

规则格式示例：
```json
{
  "permissions": {
    "allow": ["Bash(npm test:*)", "FileRead"],
    "deny": ["Bash(rm -rf:*)"],
    "ask": ["Bash(npm publish:*)"]
  }
}
```

## REPL 交互流程

当决策为 `ask` 且在交互式 REPL 中：

```
useCanUseTool() Hook
  │
  ├── createPermissionContext()    → 创建权限上下文
  │   └── resolveOnce             → 保证只 resolve 一次（防并发竞争）
  │
  ├── hasPermissionsToUseTool()   → 获取决策
  │
  └── 决策为 ask？
      │
      ├── handleCoordinatorPermission()   → Coordinator 模式处理
      │   └── awaitClassifierAutoApproval → 等待分类器自动批准
      │
      ├── handleSwarmWorkerPermission()   → Swarm 工作者处理
      │   └── 权限冒泡到 leader
      │
      └── handleInteractivePermission()   → 交互式处理
          │
          ├── pushToQueue(ToolUseConfirm)  → 加入 UI 渲染队列
          │
          ├── 用户选择：
          │   ├── Allow（一次）    → resolve(allow)
          │   ├── Allow Always   → persist 规则 + resolve(allow)
          │   ├── Deny           → resolve(deny) + 可附带原因
          │   └── Abort (Ctrl+C) → resolve(deny) + AbortError
          │
          └── removeFromQueue()
```

## 安全检查（bypass 免疫）

某些路径被视为"安全敏感"，即使在 `bypassPermissions` 模式下也需要确认：

- `.git/` 目录下的文件
- `.claude/` 目录下的文件
- `.vscode/` 配置文件
- Shell 配置文件（`.bashrc`, `.zshrc` 等）

这通过 `decisionReason.type === 'safetyCheck'` 标识，在步骤 1g 和 auto 模式中被特殊处理。

## 拒绝跟踪与回退

Auto 模式维护一个拒绝追踪状态（`DenialTrackingState`）：

```typescript
type DenialTrackingState = {
  consecutiveDenials: number    // 连续拒绝次数
  totalDenials: number          // 总拒绝次数
  lastDenialTime: number        // 最后拒绝时间
}
```

- 任何工具 allow → 重置 `consecutiveDenials`
- 连续拒绝超过阈值 → `shouldFallbackToPrompting()` → 回退为交互式 ask
- 防止 AI 陷入"无限被拒绝"的死循环

## 设计亮点

### 1. 防御优先（Deny > Allow > Ask）
判决链先检查所有否定条件（deny → safetyCheck），再检查肯定条件（bypass → allow），最后兜底为 ask。fail-closed 思想。

### 2. 权限冒泡（bubble）
子 Agent 的权限请求可以"冒泡"到父进程的终端弹窗，通过 `permissionMode: 'bubble'` 实现。Fork 子代理默认使用此模式。

### 3. resolveOnce 竞争防护
`PermissionContext` 使用 `claim()` 原子操作：多个异步回调（分类器、Hook、用户）竞争同一个 resolve，只有第一个成功。

### 4. 渐进式权限
用户可以在 ask 弹窗中选择 "Allow Always"，规则持久化到配置文件。会话积累的规则逐步减少弹窗频率。

### 5. Sandbox 自动放行
当 Bash 命令运行在 Sandbox 中时，即使有 ask 规则也可以自动放行（`autoAllowBashIfSandboxed`），因为 Sandbox 本身提供了隔离保护。
