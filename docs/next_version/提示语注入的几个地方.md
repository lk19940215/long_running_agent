# 如何在当前项目内结合提示语以及 Hook 注入确保模型按规则产出

> 调研背景：`claude-coder plan` 命令生成计划时，模型确实写出了计划文件（Write 到 `~/.claude/plans/*.md`），但 harness 侧 `extractPlanPath` 提取路径失败，导致 `[ERROR] 计划生成失败: no_path`。
> 核心问题：如何让模型**稳定地按预期格式产出**，以及 harness 如何**可靠地捕获关键产物**。

---

## 一、问题根因分析

### 1.1 当前路径提取链路

```
buildPlanOnlyPrompt()     →  提示模型输出路径
                               ↓
sdk.query(plan mode)      →  模型执行：探索 → 交互 → Write 计划 → ExitPlanMode
                               ↓
extractResultText()       →  从 collected 中找 type==='result' 的 msg.result
                               ↓
extractPlanPath(result)   →  正则匹配 `.md` 路径
                               ↓
fs.existsSync(planPath)   →  验证文件存在
```

### 1.2 失败点逐项分析

| 编号 | 失败点 | 日志证据 | 根因 |
|------|--------|----------|------|
| F1 | `result.result` 为空 | plan-log-1/2/3 全部 `no_path` | ExitPlanMode 后 SDK 等待用户审批，30s 超时中断，`result` 消息可能未正常发出 |
| F2 | 模型输出路径但 extractPlanPath 匹配失败 | plan-log-3: 模型输出了 `方案文件：\`...\`` 但仍失败 | 路径文本出现在 assistant 消息中而非 result.result 中；extractResultText 只取 result 类型 |
| F3 | 正则匹配到错误路径 | plan-log.log (首页案例): 模型输出 `.tsx` 路径 | 正则 `/`([^`]+\.md)`/` 过于宽泛，可能匹配非计划文件路径 |
| F4 | ExitPlanMode 不触发 PostToolUse hook | SDK 设计限制 | ExitPlanMode/EnterPlanMode 是特殊工具，不触发 PreToolUse/PostToolUse |

### 1.3 核心矛盾

**提示语要求模型在文本中输出路径** vs **模型的文本输出不在 `result.result` 中而在 `assistant` 消息流中**。

plan mode 下模型调用 ExitPlanMode 后，SDK 进入审批等待流程。此时 `result` 消息的 `result` 字段可能为空或不包含模型之前输出的文本。harness 侧仅从 `result.result` 提取路径，导致丢失。

---

## 二、三层防御方案

### 核心思路：不依赖模型文本输出，直接从工具调用中捕获路径

```
第一层：PostToolUse Hook 拦截 Write 调用（最可靠）
第二层：从 assistant 消息流中提取路径（备选）
第三层：提示语优化 + ExitPlanMode 参数解析（兜底）
```

### 2.1 第一层：PostToolUse Hook 拦截 Write（推荐，100% 可靠）

**原理**：plan mode 下模型使用 Write 工具写入计划文件到 `~/.claude/plans/`。Write 是普通工具，PostToolUse hook **一定会触发**。直接从 `tool_input.file_path` 中获取路径，无需依赖模型文本输出。

**实现**：

```javascript
// hooks.js - 新增 plan path capture module
function createPlanPathCaptureModule() {
  const plansDir = path.join(os.homedir(), '.claude', 'plans');
  let capturedPath = null;

  return {
    preHook: async (input) => {
      // PreToolUse: 可在 Write 之前注入提示（可选）
      return {};
    },
    postHook: async (input) => {
      // PostToolUse: 拦截 Write 调用，捕获计划文件路径
      if (input.tool_name === 'Write') {
        const target = input.tool_input?.file_path || input.tool_input?.path || '';
        if (target.startsWith(plansDir) && target.endsWith('.md')) {
          capturedPath = target;
        }
      }
      return {};
    },
    getPath: () => capturedPath,
  };
}
```

**优势**：
- 不依赖模型文本输出，不受 ExitPlanMode 审批流程影响
- 不依赖 `result.result` 内容
- 路径来自 Write 工具的实际参数，100% 准确
- 即使模型不按格式输出路径，只要它写了文件就能捕获

### 2.2 第二层：从 assistant 消息流中提取路径

**原理**：所有日志都显示模型在 assistant 消息中输出了路径文本，只是 `result.result` 为空。从 collected 消息中提取全部 assistant 文本作为备选。

```javascript
function extractPlanPathFromMessages(collected) {
  // 优先从 result.result 提取
  const resultText = extractResultText(collected);
  if (resultText) {
    const path = extractPlanPath(resultText);
    if (path && fs.existsSync(path)) return path;
  }

  // 备选：从所有 assistant 文本中提取
  let fullText = '';
  for (const msg of collected) {
    if (msg.type === 'assistant' && msg.message?.content) {
      for (const block of msg.message.content) {
        if (block.type === 'text' && block.text) {
          fullText += block.text;
        }
      }
    }
  }
  return extractPlanPath(fullText);
}
```

### 2.3 第三层：ExitPlanMode 参数解析

**原理**：模型调用 ExitPlanMode 时传入 `plan` 参数（完整计划文本）。虽然 ExitPlanMode 不触发 hooks，但其 `tool_use` block 出现在 assistant 消息中，可以从中提取信息。

```javascript
// 在消息循环中，已有 ExitPlanMode 检测逻辑
if (block.type === 'tool_use' && block.name === 'ExitPlanMode') {
  exitPlanModeDetected = true;
  exitPlanModeTime = Date.now();
  // 新增：从 ExitPlanMode 的 input.plan 中提取文件路径
  if (block.input?.plan) {
    const planText = block.input.plan;
    // 计划文本中可能包含路径引用
  }
}
```

## 三、提示语优化策略

### 3.1 当前提示语问题

```javascript
// plan.js - buildPlanOnlyPrompt
return `${userInput}
${constraint}
【重要】在最后输出中，必须包含实际方案文件的写入路径，格式如下：
方案文件：\`<实际路径>\`
`;
```

**问题**：
1. 提示在 prompt 末尾，容易被模型后续工具调用和思考覆盖
2. "最后输出"含义模糊——模型认为 ExitPlanMode 前的文本就是"最后输出"，但 harness 期望的是 `result.result`
3. 模型可能在多处输出 `.md` 路径，正则匹配歧义

### 3.2 优化后的提示语

```javascript
function buildPlanOnlyPrompt(userInput, interactive = false) {
  const constraint = interactive
    ? '【约束】如果有不确定的关键决策点，请使用 AskUserQuestion 工具向用户提问。'
    : '【约束】不要提问，默认使用最佳推荐方案。';

  return `${userInput}
${constraint}

【工作流程】
1. 探索项目代码库，理解结构和技术栈
2. ${interactive ? '通过 AskUserQuestion 确认关键决策' : '使用最佳推荐方案'}
3. 使用 Write 工具将完整计划写入 ~/.claude/plans/ 目录（.md 格式）
4. 在写入计划文件后，立即输出以下标记行（独占一行，不要嵌入其他文本中）：

PLAN_FILE_PATH: <你刚写入的计划文件绝对路径>

5. 简要总结计划要点
`;
}
```

**改进点**：
- 使用明确的工作流程步骤，而非模糊的"最后输出"
- 用 `PLAN_FILE_PATH:` 标记替代反引号包裹，避免 markdown 格式干扰
- 标记行要求独占一行，便于精确匹配

### 3.3 对应的正则优化

```javascript
function extractPlanPath(text) {
  // 优先匹配结构化标记
  const tagMatch = text.match(/PLAN_FILE_PATH:\s*(\S+\.md)/);
  if (tagMatch) return tagMatch[1];

  // 备选：匹配 ~/.claude/plans/ 下的路径
  const plansDir = path.join(os.homedir(), '.claude', 'plans');
  const escaped = plansDir.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
  const dirMatch = text.match(new RegExp(`(${escaped}/[^\\s\`']+\\.md)`));
  if (dirMatch) return dirMatch[1];

  // 最终兜底
  const fallback = text.match(/`([^`]+\.md)`/) || text.match(/\/[^\s`']+\.md/);
  return fallback ? (fallback[1] || fallback[0]) : null;
}
```

## 四、Hook 注入提示语的时机与策略

### 4.1 可用的注入时机

| 时机 | Hook 事件 | 可返回字段 | 适用场景 |
|------|-----------|------------|----------|
| 工具调用前 | PreToolUse | `additionalContext`, `systemMessage` | 工具级操作指南 |
| 工具调用后 | PostToolUse | 无注入能力，但可读取结果 | 状态捕获、路径记录 |
| Prompt 提交时 | UserPromptSubmit | `systemMessage` | Session 级全局规则 |
| Session 开始 | SDK systemPrompt | N/A (直接参数) | 角色定义、全局约束 |

### 4.2 针对 Plan 场景的 Hook 注入策略

#### 策略 A：ExitPlanMode 前的 Write 调用注入提醒（PreToolUse）

```javascript
// 当模型调用 Write 写入 plans 目录时，提醒它输出路径标记
{
  name: "plan-write-reminder",
  matcher: "Write",
  condition: {
    field: "tool_input.file_path",
    pattern: "\\.claude/plans/.*\\.md$"
  },
  // 通过 additionalContext 注入
  tip: "文件写入后，请在下一条文本消息中独占一行输出：PLAN_FILE_PATH: <文件绝对路径>"
}
```

#### 策略 B：通过 guidance.json 配置（零代码扩展）

```json
{
  "name": "plan-path-output",
  "matcher": "Write",
  "condition": {
    "field": "tool_input.file_path",
    "pattern": "\\.claude/plans/.*\\.md$"
  },
  "toolTips": {
    "injectOnce": false,
    "extractor": "(Write)",
    "items": {
      "Write": "计划文件写入成功后，请在下一条输出中独占一行标注：PLAN_FILE_PATH: <刚写入的文件绝对路径>"
    }
  }
}
```

> 注意：当前 plan session 未启用 guidance 模块。需要在 FEATURE_MAP 中为 plan/plan_interactive 添加 GUIDANCE 功能。

#### 策略 C：systemPrompt 注入全局角色约束

```javascript
// plan.js - _executePlanGen
const queryOpts = {
  permissionMode: 'plan',
  cwd: opts.projectRoot || assets.projectRoot,
  hooks: ctx.hooks,
  systemPrompt: '你是一个技术方案规划专家。在完成方案写入后，必须输出 PLAN_FILE_PATH: <路径> 标记行。',
};
```

### 4.3 推荐的组合策略

```
提示语层：buildPlanOnlyPrompt 中明确工作流程和输出格式
     ↓
Hook 层：PostToolUse 拦截 Write 调用，直接捕获路径（第一层防御）
     ↓
Hook 层：PreToolUse 在 Write plans/*.md 时注入提醒（第二层加固）
     ↓
代码层：extractPlanPath 从 assistant 消息流提取（第三层兜底）
```

## 五、Plan 模式下的 FEATURE_MAP 扩展

### 5.1 当前映射

```javascript
FEATURE_MAP = {
  plan:             [STALL],
  plan_interactive: [STALL, INTERACTION],
}
```

Plan session 目前只有 stall 检测和交互支持，没有 guidance 注入、没有 PostToolUse 处理。

### 5.2 建议扩展

```javascript
FEATURE_MAP = {
  plan:             [STALL, GUIDANCE, PLAN_CAPTURE],
  plan_interactive: [STALL, INTERACTION, GUIDANCE, PLAN_CAPTURE],
}
```

新增模块：
- `GUIDANCE`：启用 guidance.json 规则，在 Write plans 文件时注入路径输出提醒
- `PLAN_CAPTURE`：PostToolUse hook 捕获 Write 到 plans 目录的路径

## 六、关于 `result.result` 为空的根因与应对

### 6.1 SDK 行为分析

```
模型执行流程（plan mode）：
  assistant: 文本输出 + Write 调用
  assistant: 文本输出 + ExitPlanMode 调用
  SDK: 等待用户审批（CLI 模式下弹出 permission dialog）
  harness: 30s 超时 → 中断循环 → result 消息未发出
```

在 harness 的 headless 环境中，ExitPlanMode 触发的审批流程无人响应，导致 SDK 不会发出 `result` 消息。harness 的 30s 超时逻辑打断了循环，但此时 collected 中没有 `type === 'result'` 的消息。

### 6.2 关键发现

从日志分析，模型**每次都**成功完成了以下步骤：
1. 探索代码库
2. 与用户交互（interactive 模式）
3. 使用 Write 写入计划文件到 `~/.claude/plans/`
4. 在 assistant 文本中输出了路径
5. 调用 ExitPlanMode

失败点**全部**在步骤 5 之后——harness 无法从 ExitPlanMode 审批流程中获取 `result.result`。

### 6.3 根本解决

**不再依赖 `result.result`**。使用 PostToolUse Hook 在步骤 3（Write）时直接捕获路径。这样即使 ExitPlanMode 超时，路径也已经被记录。

## 八、通用经验：确保模型按规则产出的方法论

### 8.1 可靠性金字塔

```
         ┌───────────────────┐
         │  提示语约束       │  ← 最不可靠，模型可能忽略
         │  (Prompt Layer)   │
         ├───────────────────┤
         │  Hook 注入提醒    │  ← 工具调用时即时注入，上下文更强
         │  (Injection Layer)│
         ├───────────────────┤
         │  Hook 数据捕获    │  ← 直接从工具参数获取，不依赖模型输出
         │  (Capture Layer)  │
         ├───────────────────┤
         │  多源提取兜底     │  ← 从 result、assistant、工具参数多源提取
         │  (Extraction Layer│
         └───────────────────┘
              最可靠
```

### 8.2 关键原则

1. **不信任模型文本输出**：模型的文本输出（`result.result`、assistant text）是"尽力而为"的，可能缺失、格式错误、或被其他内容覆盖。关键数据应从工具调用参数（`tool_input`）中直接获取。

2. **Hook 是最可靠的观测点**：PreToolUse/PostToolUse 在每次工具调用时触发，参数结构固定。从 `tool_input` 中提取数据比从自然语言输出中正则匹配可靠得多。

3. **提示语是引导而非保障**：提示语可以提高模型按期望格式输出的概率，但不能保证 100% 遵守。应将提示语视为"第一层引导"，用 Hook 作为"硬保障"。

4. **多源提取、逐级降级**：关键数据的提取应有多层来源——Hook 捕获 > assistant 消息提取 > result.result 提取。前层成功则跳过后层。

5. **ExitPlanMode 是特殊路径**：plan mode 下 ExitPlanMode 不触发 hooks，且可能导致 SDK 进入等待审批状态。harness 必须在 ExitPlanMode 之前完成关键数据的捕获。

### 8.3 适用于其他场景的模式

| 场景 | 提示语层 | Hook 注入层 | Hook 捕获层 |
|------|----------|------------|------------|
| Plan 路径提取 | 工作流 + 标记行 | Write plans 时提醒 | PostToolUse 拦截 Write |
| Session Result 提取 | agentProtocol 约束格式 | — | PostToolUse 检测 session_result 写入 |
| 任务分解格式 | addGuide.md 模板 | — | 可扩展 PostToolUse 验证 tasks.json 格式 |
| 代码质量约束 | coding 提示语 | guidance.json 编辑提醒 | editGuard 频率检测 |
