## 1. 总体评价

### 重构质量：7.5 / 10

重构将原先扁平的 `src/*.js` 结构迁移为三层分层架构，整体方向正确、组织合理：

```
src/
├── common/    # 共享基础设施（7 个文件）
│   ├── config.js, constants.js, utils.js
│   ├── logging.js, indicator.js, sdk.js, tasks.js
├── core/      # 核心运行时（12 个文件）
│   ├── runner.js, validator.js, coding.js
│   ├── base.js, context.js, hooks.js
│   ├── prompts.js, scan.js, init.js
│   ├── plan.js, simplify.js, utils.js
└── commands/  # CLI 命令（2 + 6 个文件）
    ├── auth.js, setup.js
    └── setup-modules/ (6 个子模块)
```

**优点**：
- 常量集中管理（`constants.js`）——状态机、超时、文件名统一维护
- 工具函数抽取（`common/utils.js`）——`readJson`、`writeJson`、`getGitHead` 等统一封装
- Session 执行器模式（`base.js` + `context.js`）——`runSession()` 提供统一生命周期管理
- Hook 工厂模式（`hooks.js`）——按 session 类型自动组装 PreToolUse/PostToolUse 链
- 模板引擎分离（`prompts.js` + `templates/`）——静态文本从代码中剥离

**问题**：
- 文档完全没有跟上重构（目录结构、命令列表、模块路径大面积失效）
- 存在 3 个运行时 Bug（可能导致崩溃）
- `package.json` 配置有遗留错误
- 测试覆盖严重不足（核心校验和主循环无测试）

## 2. 致命问题（必须修复）

### P0-1: `simplify` 命令 API 签名不匹配

**文件**：`bin/cli.js` 第 129-131 行 + `src/core/simplify.js` 第 48 行

CLI 将 `positional[0]`（字符串）作为第一个参数传入：

```javascript
// bin/cli.js:129-131
case 'simplify': {
  const { simplify } = require('../src/core/simplify');
  await simplify(positional[0] || null, { n: opts.n });
  break;
}
```

但 `simplify(n, opts)` 期望 `n` 为数字（commit 数量）：

```javascript
// src/core/simplify.js:11
async function _runSimplifySession(n = 3, opts = {}) {
  // ...
  diff = execSync(`git diff HEAD~${n}..HEAD`, ...);
```

当用户执行 `claude-coder simplify "内存效率"` 时，实际运行的是 `git diff HEAD~内存效率..HEAD`，必定失败。

**修复建议**：调整 `simplify` 函数签名为 `simplify(focus, opts)`，从 `opts.n` 取 commit 数量，`focus` 作为审查聚焦方向注入 prompt。

---

### P0-2: `prompts.js` 中 `depends_on` 未防御

**文件**：`src/core/prompts.js` 第 260 行

```javascript
recent.map(f => `...depends_on=[${f.depends_on.join(',')}])`).join('\n');
```

当任务缺少 `depends_on` 字段时，`undefined.join()` 抛出 `TypeError: Cannot read properties of undefined`。

**修复建议**：改为 `(f.depends_on || []).join(',')`。

---

### P0-3: `init.js` 的 `loadProfile()` 无异常防护

**文件**：`src/core/init.js` 第 11-13 行

```javascript
function loadProfile() {
  const p = paths();
  return JSON.parse(fs.readFileSync(p.profile, 'utf8'));
}
```

若 `project_profile.json` 损坏（空文件、截断 JSON），`JSON.parse` 直接抛异常，调用方（第 100 行 `const profile = loadProfile()`）无 try/catch，进程崩溃。

**修复建议**：使用 `readJson(p.profile, null)` 并在调用处处理 null。

---

### P0-4: `scan.js` 的 `validateProfile()` 同样无异常防护

**文件**：`src/core/scan.js` 第 19 行

```javascript
const profile = JSON.parse(fs.readFileSync(p.profile, 'utf8'));
```

与 P0-3 相同的问题模式。

**修复建议**：使用 `readJson()`。

---

### P0-5: `package.json` 的 `files` 字段包含不存在的目录

**文件**：`package.json` 第 8-13 行

```json
"files": [
  "bin/",
  "src/",
  "prompts/",   // ← 不存在，应删除
  "templates/"

```

`prompts/` 目录不存在，是重构遗留。虽然 npm 打包时会忽略不存在的条目，但这是配置错误。

**修复建议**：删除 `"prompts/"`。

## 5. 代码异味与优化建议

### 5.1 `getTemplatePath` 和 `getPromptPath` 实现完全相同

**文件**：`src/common/config.js` 第 42-48 行

```javascript
function getTemplatePath(name) {
  return path.join(__dirname, '..', '..', 'templates', name);
}

function getPromptPath(name) {
  return path.join(__dirname, '..', '..', 'templates', name);
}
```

两个函数体完全一致，都指向 `templates/` 目录。虽然语义上区分"模板文件"和"prompt 文件"，但实际都存放在同一目录。

**建议**：保留一个（`getTemplatePath`），让另一个代理到它：`const getPromptPath = getTemplatePath;`。或合并为一个函数。

### 5.2 `log()` 函数放在 `config.js` 而非 `logging.js`

`src/common/config.js` 导出 `log(level, msg)`，所有模块都从 config 导入它。但 `src/common/logging.js` 已经是专门的日志模块（处理 SDK 消息日志）。

简单的控制台日志函数放在 config 模块中是反直觉的，但因为被 27 个文件广泛引用，重构成本较高。

**建议**：保持现状，但在 config.js 顶部加注释说明 `log` 的存在理由（避免循环依赖）。

### 5.3 两个 `utils.js` 命名容易混淆

- `src/common/utils.js` — JSON 读写、Git 工具、字符串截断、sleep
- `src/core/utils.js` — `hasCodeFiles()`、`buildQueryOptions()`

两个同名文件在不同目录，虽然职责清晰（一个通用、一个 core 专用），但在 import 时可能困惑。

**建议**：将 `src/core/utils.js` 更名为 `src/core/query.js` 或 `src/core/sdk-helpers.js`，更好地表达其职责（SDK 查询选项构建 + 项目检测）。

### 5.4 `JSON.parse(fs.readFileSync)` 模式散落

以下位置直接使用 `JSON.parse(fs.readFileSync(...))` 而非已有的 `readJson()` 工具函数：

| 文件 | 行号 | 建议 |
|------|------|------|
| `src/core/init.js` | 13 | 改用 `readJson()` |
| `src/core/scan.js` | 19 | 改用 `readJson()` |
| `src/core/prompts.js` | 130, 142, 232 | 已包在 try/catch 中，可改用 `readJson()` 简化 |
| `src/core/validator.js` | 123 | `checkTestCoverage` 中已包在 try/catch 中 |

### 5.5 `runner.js` 中 `loadConfig()` 被调用两次

第 146 行（循环外）和第 253 行（循环内）各调用一次 `loadConfig()`。循环内的调用是为了读取 `simplifyInterval` 配置，但每轮都重新解析 `.env` 文件。

**建议**：在循环外调用一次并缓存结果。如果需要支持运行时配置热更新，在注释中说明。

### 5.6 `context.js` 构造函数 `process.env` 副作用

`SessionContext` 构造函数中 `this._applyEnvConfig()` 直接 `Object.assign(process.env, ...)` 修改全局环境变量。这在单进程中没问题，但会让测试变得困难。

## 6. Git/File 操作抽离分析

### 6.1 当前散布情况

| 操作类别 | 涉及文件 | 具体操作 |
|----------|----------|----------|
| Git 状态查询 | `validator.js` | `execSync('git rev-parse HEAD')`, `execSync('git log --oneline -1')` |
| Git 生命周期 | `runner.js` | `git init`, `git add`, `git commit`, `git reset --hard`, `git push`, `git diff` |
| Git 审查 | `simplify.js` | `git diff HEAD~N..HEAD` |
| Git 工具函数 | `common/utils.js` | `getGitHead()`, `isGitRepo()` |
| 文件部署 | `init.js` | `deployFile()`, `deployGuidanceFiles()` |
| .gitignore | `auth.js` + `setup-modules/helpers.js` | 两处独立实现的 gitignore 追加逻辑 |

### 6.2 抽离建议

**不需要独立 `git.js` 模块**。理由：
- Git 操作总量不大（~30 行散布在 runner + validator + simplify）
- `runner.js` 中的 Git 操作（回滚、推送、初始化）与业务逻辑强耦合，拆出去反而增加间接层
- `common/utils.js` 已提供了 `getGitHead()` 和 `isGitRepo()`

**应该做的**：
1. `validator.js` 的 `checkGitProgress()` 应复用 `getGitHead(cwd)` 而非直接 `execSync`
2. `.gitignore` 操作应统一为一个共享函数（当前 `auth.js` 和 `helpers.js` 各有一套）
3. 散落的 `JSON.parse(fs.readFileSync)` 应统一使用 `readJson()`
