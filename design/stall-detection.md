# 停顿检测与会话中断策略

## 决策：使用 AbortController.abort() 而非 session.send()

### 背景

在长时间运行的 harness 中，模型可能因以下原因停止响应：

1. **正常完成后未退出**：模型已写入 `session_result.json`，但未终止会话
2. **真正卡住**：模型陷入死循环或等待不存在的响应

需要一种可靠的机制检测并中断这类情况。

### 方案对比

#### AbortController.abort()（当前方案）

- **机制**：通过 `AbortController` 向 SDK 的异步迭代器发送取消信号，强制终止当前会话
- **行为**：确定性硬停止，SDK 立即中断 API 请求
- **适用 API**：V1 `sdk.query()` 原生支持

#### session.send()（V2 SDK）

- **机制**：向活跃会话追加一条用户消息，模型继续处理
- **行为**：非确定性软提示，依赖模型理解并配合"请停止"指令
- **适用 API**：V2 `unstable_v2_createSession`（不稳定接口）

### 决策依据

| 维度           | abort()              | send()                  |
| -------------- | -------------------- | ----------------------- |
| 终止可靠性     | 确定性，立即生效     | 非确定性，依赖模型配合  |
| API 稳定性     | V1 正式接口          | V2 不稳定接口           |
| 适用场景       | 超时/卡死中断        | 多轮对话追加输入        |
| 副作用         | 会话直接终止         | 可能触发额外输出        |
| 延迟           | 毫秒级               | 需等待模型响应          |

**结论**：`abort()` 是 stall 检测场景的正确选择。`send()` 设计目的是多轮对话，不适用于超时中断。

### 当前实现

```
hooks.js: createStallModule()
├── 每 30 秒检查一次活跃度
├── 正常模式：idle > 20 分钟 → abort()
└── 完成模式：session_result 写入后 > 5 分钟 → abort()
```

活跃度由两个来源更新：
- PreToolUse hook（工具调用前）
- logMessage（模型文本输出时）

### 未来考虑

- 若 V2 SDK 稳定并支持优雅停止信号（如 `session.stop()`），可评估替代方案
- 当前架构已通过 `AbortController` 与 `_isStalled()` 双重检查保证可靠中断
